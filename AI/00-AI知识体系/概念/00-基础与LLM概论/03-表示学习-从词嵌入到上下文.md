# 表示学习：从词嵌入到上下文

> 模型不直接处理「单词」，而是处理**向量**。LLM 的强大之处在于：同一个 token 在不同上下文里会得到**不同的向量**（上下文相关表示）。

## 为什么需要 Embedding

- 神经网络输入必须是数值；**离散符号**（字、词、子词）需映射到连续向量空间
- **词嵌入（Word Embedding）**：每个 token（或子词）对应一个可学习的向量 \(e_i \in \mathbb{R}^d\)
- 语义相近的 token 在训练中倾向于**靠近**（非严格保证，但是有用直觉）

## 静态嵌入 vs 上下文嵌入

| | 静态（如 Word2Vec、GloVe） | 上下文（LLM 内部） |
|--|---------------------------|-------------------|
| 一词一向量 | 「bank」无论银行还是河岸，向量相同 | 每层 Attention + FFN 后，表示依赖**整句前文** |
| 用途 | 传统 NLP 特征、检索基线 | 现代 LLM 的默认 |

LLM 的 **Token Embedding 层**只是起点；经过多层 Transformer 后，每个位置的 hidden state 已是**上下文相关**的语义表示。

## LLM 内部的表示流水线

```text
token ID
→ token embedding
→ 加入位置信息
→ 第 1 层 hidden state
→ 第 2 层 hidden state
→ ...
→ 最后一层 hidden state
→ LM Head 转成词表上每个 token 的分数
```

同一个 token 在 embedding 表里的初始向量是固定的；但进入 Transformer 后，它会不断吸收上下文信息。比如“苹果”在“我买了苹果手机”和“我吃了苹果”里，初始 token 可能相同，后续 hidden state 会逐层变得不同。

## 维度与表大小

- **\(d_{\text{model}}\)**：隐藏维度（如 4096），决定表示容量
- **词表大小 \(V\)**：常见 32k～128k+；Embedding 矩阵规模 \(V \times d\)，占参数量可观
- **输出头（LM Head）**：常与输入嵌入**权重绑定（weight tying）**，减少参数

## 与 RAG 里「Embedding 模型」的关系

- **同一概念、不同产品**：
  - LLM 内部：为**生成**服务的上下文表示
  - RAG 检索：用专门的 **Embedding 模型**（如 BGE、text-embedding-3）把文档块编成向量，做相似度搜索
- 详见 [[AI/00-AI知识体系/概念/04-RAG进阶/03-Embedding模型与向量检索|Embedding模型与向量检索]]

## 关键直觉

1. **Token 是 LLM 的原子单位**，不是「字」或「词」；见 [[AI/00-AI知识体系/概念/00-基础与LLM概论/05-Tokenizer与分词|Tokenizer与分词]]
2. **位置信息**必须额外注入（绝对/相对/RoPE），否则 Attention 对顺序不敏感
3. **长上下文** = 在有限窗口内保留更多 token 的表示与 KV，见 [[AI/00-AI知识体系/概念/01-模型层/07-上下文与KV Cache|上下文与KV Cache]]

## 容易混淆的两种 Embedding

| 名称 | 用途 | 什么时候用 |
|---|---|---|
| **Token embedding** | LLM 内部把 token ID 变成可计算向量 | 每次模型前向传播都会用 |
| **Text embedding / RAG embedding** | 把一段文本变成一个向量，用于相似度检索 | 建索引、搜索相关文档块时用 |

RAG embedding 不会直接替换 LLM 的 token embedding；它通常在模型外部帮你找到相关文本，再把文本放进 prompt。
