# 什么是 LLM

> **Large Language Model（大语言模型）**：在海量文本上预训练、参数量达数十亿以上的**自回归语言模型**；通过 API 或本地部署，以「续写文本」的方式完成对话、推理、代码、工具调用等任务。

## 核心定义

| 维度 | 说明 |
|------|------|
| **Large** | 参数量与训练数据规模大（常见 7B～数百 B）；算力与数据是能力上限的重要来源 |
| **Language** | 建模对象是**自然语言（及代码等文本）的 token 序列** |
| **Model** | 可学习的神经网络；输入前文，输出下一 token 的概率分布 |

## LLM 在做什么（一句话）

给定 token 序列 \(x_1,\ldots,x_t\)，学习 \(P(x_{t+1} \mid x_1,\ldots,x_t)\)，推理时**逐个采样**生成后续 token，直到结束符或长度上限。

语言模型训练时学的是“根据前文预测下一个词/符号”；真正生成时，它每次只生成一个 token，把刚生成的 token 接到后面，再继续预测下一个，直到该停为止。

## 从输入到输出的完整链路

```text
用户文本
→ tokenizer 切成 token ID
→ embedding 表把 ID 映射成向量
→ Transformer 多层计算上下文表示
→ LM Head 输出所有候选 token 的 logits
→ softmax / sampling 选出下一个 token
→ 拼回上下文，继续生成
```

所以 LLM 不是直接“理解一句中文”，而是在一个巨大的向量空间里，根据上下文计算下一个 token 的概率。所谓理解、推理、写代码，是这个训练目标在足够大模型、足够多数据和合适对齐下表现出来的能力。

## 三层心智模型

| 层 | 你可以怎么理解 |
|---|---|
| **底层** | 一个神经网络函数：输入 token 序列，输出下一个 token 的概率分布 |
| **中层** | 一个被海量文本训练出的模式压缩器：学到语言、事实关联、代码结构、推理轨迹 |
| **上层** | 一个可通过 prompt / tools / RAG / agent harness 使用的通用接口 |

学习 LLM 时容易混淆这三层：模型本身只生成 token；Chat、联网、读文件、调用工具，是外层系统给它加的能力。

## 与传统 NLP / 小模型的区别

| | 传统 NLP（规则 / 小模型） | LLM |
|--|---------------------------|-----|
| 任务形式 | 常为「一个任务一个模型」（分类、NER…） | **通用**：同一模型经提示或微调适配多任务 |
| 知识来源 | 特征工程 + 标注数据 | **预训练**从语料吸收语言与常识；RAG 可补实时知识 |
| 交互方式 | 固定输入输出 schema | **自然语言指令** + 可选工具 / JSON 模式 |
| 能力涌现 | 较弱 | 规模上去后出现 ICL、推理、工具使用等（非保证） |

## 能力边界（应用工程师必知）

- **擅长**：语言理解/生成、格式遵循、模式模仿、在上下文内的推理与综合
- **不擅长 / 需外挂**：实时事实、私有数据、精确计算、可验证执行 → 用 **RAG、工具、代码解释器**
- **幻觉**：高置信的错误陈述；不能单靠「更大模型」消除，需评测与约束
- **上下文窗口**：能「看见」的 token 有上限；超长文档要分块、摘要或检索

## 常见产品形态

- **Chat API**：OpenAI、Anthropic、Gemini 等（背后多为 Decoder-only Transformer）
- **开源权重**：Llama、Qwen、DeepSeek、Mistral（可本地 [[AI/00-AI知识体系/概念/02-训练与推理/09-推理引擎与Serving|Serving]]）
- **Agent 产品**：在 LLM 外包一层 [[AI/00-AI知识体系/概念/03-Agent系统/05-Agent Harness|Harness]] + 工具

## 常见误区

- **误区 1：LLM 像数据库一样存事实。** 更准确地说，它在参数里压缩了训练数据的统计规律；事实可能过期、混淆或被上下文诱导。
	- LLM 的“知识”不是可靠数据库查询结果，而是训练数据模式的压缩表达；因此它可能不知道最新事实，可能混淆相似信息，也可能被你的上下文带偏。
	
- **误区 2：生成流畅就等于正确。** 语言概率高不代表事实正确，尤其是时间敏感、细节密集、可计算任务。
- **误区 3：模型越大，应用就越可靠。** 可靠性还依赖检索、工具、评测、权限、提示格式和失败处理。
- **误区 4：Prompt 就是全部。** 真正的 LLM 应用通常是 prompt + context + tools + eval + observability。

## 学习检查

读完这篇，你应该能解释：

1. LLM 为什么可以被看成“概率续写器”？
	1. LLM 可以被看成“概率续写器”，因为它最核心的训练目标就是： 给定前文，预测下一个 token 的概率分布。
	2. 从底层机制看，LLM 并不是一次性“想好整段答案”，而是在不断做: 看已有上下文 → 预测下一个 token → 选一个 token → 接回上下文 → 继续预测
	3. 它之所以看起来会聊天、写代码、推理，是因为训练数据和模型规模足够大，让“预测下一个 token”这个任务逼着模型学会了语言、知识、格式、代码模式和常见推理结构。
	4. 所以“概率续写器”不是贬义，而是一个底层视角：LLM 的输出，本质上是基于上下文不断采样高概率后续 token 的过程。
	
2. Chat API 和底层 next-token 生成是什么关系？

```
Chat API 是**外层接口**，next-token 生成是**底层机制**。

你调用 Chat API 时，通常传的是结构化消息：

`[ {"role": "system", "content": "你是一个严谨的助手"}, {"role": "user", "content": "解释一下 Transformer"} ]`

但模型本身并不直接“理解 JSON 消息数组”。API 服务会先把这些消息套进某种 **chat template**，变成一段 token 序列，例如概念上类似：

`<system> 你是一个严谨的助手 </system> <user> 解释一下 Transformer </user> <assistant>`

然后底层模型仍然做同一件事：

`根据已有 token，预测下一个 token`

生成一个 token 后，把它接回上下文，再预测下一个，直到遇到结束符或长度上限。

所以关系是：

`Chat API messages → chat template 格式化 → tokenizer 转成 token IDs → LLM next-token prediction → 逐 token 生成 assistant 回复 → API 再把结果包装成 response 返回`

因此：

- **Chat API** 负责角色、消息格式、工具调用、流式返回等外层协议。
- **next-token generation** 负责真正生成内容。
- system / user / assistant 这些角色，本质上会被编码进上下文，让模型学会“现在该以 assistant 身份继续写”。

```

 3. 为什么实时事实、私有数据、精确计算通常要靠 RAG 或工具？
```
因为 LLM 本体只是根据已有上下文做 next-token 生成，它的参数不是实时数据库，也不是计算器，更不是你公司私有系统的接口。

**实时事实**  
模型训练完之后，参数基本固定。今天的股价、某个产品最新价格、公司新任 CEO、刚发布的 API 文档，它未必知道。RAG 或搜索工具可以先把最新资料取回来，再放进上下文。

**私有数据**  
模型通常没见过你的本地笔记、公司文档、数据库、邮件、代码仓库。RAG 可以从私有知识库检索相关片段；工具可以访问数据库、文件系统、CRM、工单系统等。

**精确计算**  
LLM 擅长语言模式和近似推理，但不可靠地做精确运算。比如复杂数学、日期差、财务计算、代码执行结果。工具可以调用计算器、Python、SQL、代码解释器，得到可验证结果。

LLM：负责理解问题、组织语言、规划步骤
RAG：负责找相关资料
工具：负责查实时信息、访问私有系统、做精确执行

```
