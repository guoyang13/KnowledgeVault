# 阶段 1：领域建模

## 核心方法

先不要急着写 RDF / OWL。专家建模通常从 **competency questions（能力问题）** 开始。

能力问题就是这个本体未来应该能回答的问题。然后倒推出需要哪些类、关系、属性和规则。

## 最小供应链本体

```text
Class:
Supplier, Product, SKU, Warehouse, Order, OrderLine, Shipment, Carrier

Object Property:
Supplier supplies SKU
Order hasOrderLine OrderLine
OrderLine orderedSKU SKU
SKU storedIn Warehouse
Shipment delivers Order
Shipment handledBy Carrier

Data Property:
stockQty, safetyStock, promisedDate, deliveryDate, leadTime, deliveryStatus

Rule / Constraint:
stockQty < safetyStock => ReplenishmentRisk
deliveryDate > promisedDate => LateDelivery
ColdChainSKU must be handledBy ColdChainCarrier
```

## 建模时优先问的问题

1. 这是一个类，还是一个实例？
2. 这是对象之间的关系，还是对象自己的属性？
3. 这个概念是业务必须区分的，还是暂时可以作为属性？
4. 这个规则是定义性语义，还是数据质量约束？
5. 这个词是否已有行业标准或公司内部标准？

## 常见抽象判断

| 业务表达 | 建模倾向 | 原因 |
|---|---|---|
| 供应商 | Class | 一类业务对象 |
| 供应商A | Individual | 具体实体 |
| 订单金额 | DataProperty | 数值属性 |
| 订单包含 SKU | ObjectProperty | 对象间关系 |
| 冷链 SKU | Subclass 或分类属性 | 取决于是否有专门规则 |
| 缺货风险 | 派生概念或规则结果 | 通常由库存、在途、需求、安全库存计算 |

## 本阶段交付物

1. 领域术语表。
2. 类清单。
3. 关系清单。
4. 属性清单。
5. 规则 / 约束清单。
6. 与能力问题的映射。
