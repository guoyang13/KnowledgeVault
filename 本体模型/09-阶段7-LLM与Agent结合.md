# 阶段 7：与 LLM / Agent 结合

## 核心价值

LLM 擅长语言理解和生成，但容易猜错业务含义。本体可以提供稳定的语义层。

## 常见结合方式

| 用法 | 说明 |
|---|---|
| 语义路由 | 根据本体判断问题属于库存、订单、物流还是供应商 |
| Text-to-SPARQL / SQL | 用本体约束查询生成 |
| RAG 检索增强 | 用本体扩展同义词、上下位概念和关系路径 |
| Agent 工具说明 | 工具参数绑定到本体概念 |
| 输出校验 | 用 SHACL / 规则检查 Agent 输出 |
| 决策解释 | 根据图路径和规则解释结论来源 |

## 供应链 Agent 示例

```text
用户问题：哪些 SKU 有缺货风险？

1. Agent 查本体：缺货风险的定义是什么？
2. Agent 查图谱：SKU、仓库、库存、安全库存、在途库存的关系是什么？
3. Agent 调用 SQL / SPARQL / API 获取事实数据
4. Agent 调用规则或计算工具判断风险
5. Agent 输出：风险 SKU、影响订单、原因、建议动作
6. SHACL / 规则校验输出是否缺字段、是否违反业务约束
```

## 专家提醒

- 不要让 LLM 自己发明业务口径。
- 不要把所有本体全文塞进 prompt；应按任务检索相关片段。
- 复杂优化问题交给求解器，不要让 LLM 直接拍结果。
- 高价值动作必须有人审、权限控制和审计日志。

## 本阶段交付物

1. 一个本体驱动的业务问答 demo。
2. 一个 Text-to-SPARQL 或 Text-to-SQL 示例。
3. 一个 Agent 输出校验示例。
4. 一份“哪些知识进 prompt，哪些知识走工具”的边界说明。
