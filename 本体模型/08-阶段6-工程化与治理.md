# 阶段 6：工程化与治理

## 为什么需要治理

真实项目里，本体模型不是一次画完，而是长期演进。没有治理，本体会很快变成另一种“没人敢改的遗留系统”。

## 项目结构建议

```text
ontology/
  core.ttl              # 核心类和关系
  supply-chain.ttl      # 领域本体
  shapes.ttl            # SHACL 约束
  examples.ttl          # 示例实例数据
  queries/
    risk.sparql
    inventory.sparql
  docs/
    glossary.md
    competency-questions.md
```

## 命名与版本

- URI 要稳定，不要随便改。
- 类名用名词，关系名用动词或动词短语。
- 中英文可以并存，但机器标识建议统一英文。
- 每次修改本体都记录版本和变更原因。
- 不要把临时业务规则硬塞进核心本体。

## 常用工具

| 工具 | 用途 |
|---|---|
| Protégé | 本体编辑、OWL 建模 |
| WebVOWL | 本体可视化 |
| Apache Jena / Fuseki | RDF 存储、SPARQL 服务 |
| GraphDB / Stardog | 企业级 RDF 图数据库 |
| RDFLib | Python 处理 RDF |
| pySHACL | SHACL 校验 |
| ROBOT | 本体工程自动化，常见于生命科学领域 |

## 本阶段交付物

1. 一个清晰的本体项目目录。
2. 一份命名规范。
3. 一份版本变更记录。
4. 一套示例数据和查询。
5. 一套基础校验命令或脚本。
