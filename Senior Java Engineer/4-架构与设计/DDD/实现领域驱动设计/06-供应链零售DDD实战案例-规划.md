# 供应链零售DDD实战案例规划

> 状态：待展开。这个文件先保存后续处理方案，后面可以基于它新增完整案例目录。

## 为什么要做这个案例

《实现领域驱动设计》的概念比较抽象。为了真正理解 DDD，后续应该选一条真实业务链路，把领域、子域、限界上下文、上下文映射、聚合、领域事件、Repository、应用服务等概念落到熟悉的供应链/零售场景里。

第一版不做“大而全”的供应链 DDD，而是选择一条主链路：

```text
订单支付
→ 库存预占/扣减
→ 仓储创建出库单
→ 拣货复核发货
→ 售后退货/退款
```

## 处理步骤

### 1. 写业务场景

不要先画类图，也不要先想数据库表。先描述一条业务流程：

- 谁发起？
- 系统做什么？
- 状态如何变化？
- 哪些系统参与？
- 哪些地方可能失败？
- 哪些地方需要补偿？

### 2. 建通用语言表

先整理高频业务词：

- 订单
- 库存
- 预占
- 扣减
- 出库单
- 发货
- 退货
- 退款
- 逆向入库

每个词要写清楚：

- 在哪个上下文里使用？
- 具体是什么意思？
- 和别的上下文里的同名词是否同义？
- 是否需要翻译或防腐层？

### 3. 划分子域

初步子域可以这样划：

| 子域 | 类型 | 说明 |
|---|---|---|
| 交易订单 | 核心/支撑 | 管理订单、支付、取消 |
| 库存 | 核心 | 管理可售、预占、扣减、释放 |
| 仓储履约 | 核心/支撑 | 出库、拣货、复核、发货 |
| 售后 | 支撑 | 退货、退款、补偿 |
| 通知 | 通用 | 发消息 |

### 4. 划分限界上下文

重点判断同一个词是否有不同含义。比如“订单”：

| 上下文 | 订单含义 |
|---|---|
| 交易上下文 | 买卖契约，关注金额、支付、取消 |
| 仓储上下文 | 履约任务来源，关注出库、拣货、发货 |
| 售后上下文 | 退款退货依据，关注责任和补偿 |

这一步用于帮助理解 DDD 的边界思想。

### 5. 画上下文映射

初步关系：

```text
交易上下文 --OrderPaid事件--> 库存上下文
库存上下文 --InventoryReserved事件--> 仓储履约上下文
仓储履约上下文 --ShipmentCompleted事件--> 交易上下文
售后上下文 --ReturnApproved事件--> 仓储履约上下文
```

后续需要补充：

- 谁是上游？
- 谁是下游？
- 使用 API、事件还是批处理？
- 是否需要防腐层？
- 失败如何补偿？

### 6. 设计聚合

初步聚合候选：

| 上下文 | 聚合 | 保护的不变量 |
|---|---|---|
| 交易 | Order | 已支付订单不能随意改商品 |
| 库存 | InventoryReservation | 预占数量不能超过可用数量 |
| 仓储 | ShipmentOrder | 已发货出库单不能取消 |
| 售后 | ReturnOrder | 已退款退货单不能重复退款 |

### 7. 设计领域事件

候选事件：

```text
OrderPaid
InventoryReserved
InventoryReserveFailed
ShipmentCreated
ShipmentCompleted
ReturnRequested
ReturnApproved
RefundCompleted
```

后续需要补充：

- 事件来源聚合
- 事件消费者
- 是否对外发布
- 幂等键
- 失败重试策略
- 是否需要事件版本

### 8. 落到Java包结构

初步包结构：

```text
com.example.supplychain
  trade
  inventory
  warehouse
  aftersale
  notification
```

每个上下文内部再分：

```text
application
domain
infrastructure
interfaces
```

## 最小可行版本

第一阶段只做一条链路：

```text
订单支付
→ 库存预占
→ 创建出库单
→ 仓储发货
```

先把这一条链路建模清楚，再扩到售后、退货、退款。这样 DDD 不会停留在抽象概念，会直接贴到熟悉的业务上。
