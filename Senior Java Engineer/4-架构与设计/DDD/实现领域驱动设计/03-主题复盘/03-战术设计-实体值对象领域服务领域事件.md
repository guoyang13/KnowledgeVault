## 先记住选择顺序

建模时不要默认先建实体。初学者更好的顺序是：

1. 这个概念是否只是一个值？优先考虑值对象。
2. 它是否有稳定身份和生命周期？再考虑实体。
3. 这个行为能否放进实体或值对象？能放就不要急着建领域服务。
4. 这件事发生后是否需要通知其他模型？再考虑领域事件。

## 实体

实体的重点是身份，而不是字段多。

适合建成实体的信号：

- 它有稳定身份。
- 它会经历状态变化。
- 两个对象属性相同，也可能不是同一个业务对象。
- 业务会追踪它的生命周期。

例子：

- 用户、订单、合同、项目、任务。

错误倾向：

- 为每张表都建实体。
- 实体只有 getter/setter，没有业务方法。
- 把本来可以是值对象的概念也建成实体。

## 值对象

值对象的重点是值、不可变、整体替换。

适合建成值对象的信号：

- 没有独立身份。
- 只要值相同，就可以认为相同。
- 修改时更适合整体替换，而不是改变内部字段。
- 可以表达一个业务概念，而不是散落的几个基础字段。

例子：

- 金额 Money、地址 Address、时间范围 DateRange、坐标 Coordinate、手机号 PhoneNumber。

Java 实现提醒：

```java
public record Money(BigDecimal amount, String currency) {
    public Money {
        if (amount == null || currency == null) {
            throw new IllegalArgumentException("amount and currency are required");
        }
        if (amount.signum() < 0) {
            throw new IllegalArgumentException("amount must be non-negative");
        }
    }
}
```

## 领域服务

领域服务承载领域行为，但这个行为不自然属于某个实体或值对象。

适合建领域服务的信号：

- 操作涉及多个聚合或多个领域对象。
- 行为是业务语言的一部分。
- 放到任何一个实体里都会显得牵强。
- 它通常是无状态的。

不要把这些误认为领域服务：

- 调数据库的 DAO。
- 发 HTTP 请求的 Client。
- 只做流程编排的应用服务。
- 没有业务含义的工具类。

## 领域事件

领域事件表达业务上已经发生的重要事实。

命名通常用过去式：

- `OrderCreated`
- `PaymentCompleted`
- `InventoryReserved`
- `TaskAssigned`

适合建领域事件的信号：

- 事件发生后，其他上下文或模块需要响应。
- 业务人员也关心这件事。
- 这件事可以作为审计、通知、集成或后续流程的触发点。

## 对照表

| 问题 | 更可能的建模 |
|---|---|
| 是否有身份和生命周期？ | 实体 |
| 是否只靠值判断相等？ | 值对象 |
| 是否是不属于单个对象的领域行为？ | 领域服务 |
| 是否是已经发生、值得传播的事实？ | 领域事件 |
| 是否只是技术调用？ | 基础设施服务 |
| 是否只是用例编排？ | 应用服务 |

## 阅读检查

- 我是否能把一个贫血实体改成有行为的实体？
- 我是否能从实体里抽出至少一个值对象？
- 我是否能区分领域服务和应用服务？
- 我是否能把一个状态变化命名成领域事件？
