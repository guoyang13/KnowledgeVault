## 为什么先学战略设计

初学 DDD 时，最容易直接跳到实体、值对象、资源库。但真正决定模型质量的，往往是更前面的边界问题：哪个业务最核心？同一个词在不同团队里是不是同一个意思？两个系统集成时，谁迁就谁，谁保护谁？

战略设计解决的就是这些问题。

## 领域和子域

| 类型 | 含义 | 例子 |
|---|---|---|
| 核心域 | 决定业务差异化的能力 | 电商的定价策略、物流平台的调度算法 |
| 支撑子域 | 支持核心域运转，但不是竞争力本身 | 结算、通知、报表 |
| 通用子域 | 很多公司都类似，可购买或复用 | 身份认证、文件存储、短信网关 |

判断核心域时，不要问“技术难不难”，要问“它是否决定业务成败”。

## 限界上下文

限界上下文是一套模型语言生效的边界。边界内，同一个词只能有一个清晰含义；边界外，同一个词可以有不同含义。

例如“订单”：

- 交易上下文：订单是买卖契约，关注价格、优惠、支付。
- 仓储上下文：订单是履约任务来源，关注拣货、出库、库存占用。
- 售后上下文：订单是退款、退货、补偿的依据。

这些都叫订单，但不是同一个模型。

## 通用语言

通用语言不是术语表装饰品，而是团队协作协议。它应该出现在：

- 需求讨论
- 用户故事
- 领域模型
- 类名、方法名、包名
- 测试用例
- 领域事件名称

如果业务说“冻结库存”，代码叫 `lockStock`，数据库叫 `occupy_qty`，接口叫 `reserve`，团队就需要确认这些词是否真的表达同一个概念。

## 上下文映射

上下文映射描述多个限界上下文之间的关系。常见关系如下：

| 关系 | 适用场景 | 设计提醒 |
|---|---|---|
| 合作关系 | 两个团队目标一致，频繁协同 | 需要共同规划接口和节奏 |
| 共享内核 | 少量模型必须共享 | 共享范围越小越好 |
| 客户方-供应方 | 下游依赖上游能力 | 明确上游承诺和下游诉求 |
| 遵奉者 | 下游接受上游模型 | 适合上游强势且模型可接受的场景 |
| 防腐层 | 外部模型会污染本地模型 | 用翻译层保护核心域语言 |
| 开放主机服务 | 上下文对外提供稳定协议 | 适合多个下游接入 |
| 发布语言 | 用公开格式交换信息 | 事件、DTO、API Schema 都可能承担这个角色 |
| 各行其道 | 无需集成或集成成本过高 | 接受重复实现，换取独立演进 |
| 大泥球 | 边界混乱、模型污染严重 | 识别风险，逐步隔离 |
