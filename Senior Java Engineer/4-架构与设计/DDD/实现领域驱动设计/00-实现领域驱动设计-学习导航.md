# 实现领域驱动设计学习导航

> 学习目标：把 DDD 学成一种“围绕业务语言控制软件复杂度”的方法，而不是只会背实体、值对象、聚合、资源库这些名词。

## 资料来源

- 书籍：Vaughn Vernon《实现领域驱动设计》
- 本地 PDF：`/Users/guoyang/Downloads/实现领域驱动设计 (Vaughn Vernon) (z-library.sk, 1lib.sk, z-lib.sk).pdf`
- OCR 信息：扫描版 PDF，共 572 页；已抽取前言、使用说明和部分目录页
- 阅读方式：只沉淀理解、结构、学习日志和练习，不大段搬运原文

## 初学者先抓住这句话

DDD 是一套把复杂业务软件拆成清晰模型的方法：先用业务语言划清模型边界，再在边界内用实体、值对象、领域服务、领域事件、聚合等模式表达业务规则。

## 每次阅读固定输出

每读完一小节，都按下面 5 个问题写到 [[01-初学者阅读日志]]：

1. 这一节解决什么问题？
2. 它属于战略设计、架构设计、战术设计，还是应用集成？
3. 它反对哪种常见错误做法？
4. 如果放到 Java 项目里，会对应哪些包、类、接口、测试？
5. 我自己的项目里有没有类似问题？

## 学习时的判断原则

- 先问业务语言，再问代码结构。
- 先划限界上下文，再设计实体和值对象。
- 先找业务不变量，再决定聚合边界。
- 先保护领域模型，再接入数据库、消息、HTTP、缓存等基础设施。
- 只用实体、资源库、服务类，不代表学会了 DDD。
