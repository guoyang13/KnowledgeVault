# 第11章：工厂

> 本章目标：学会在创建复杂领域对象时封装创建规则，保护模型消费方。

## 本章在全书中的位置

第10章讲聚合边界，第11章讲聚合和复杂对象如何被正确创建。工厂不是为了套设计模式，而是为了避免创建逻辑散落。

## 一句话理解

工厂封装复杂创建过程，让调用方不用知道领域对象构造细节，同时保证创建时业务规则成立。

## 本章精读路线

读工厂这一章时，要把它和“new 一个对象”区分开。工厂真正有价值的地方是保护创建时的业务语义。

重点判断：

- 创建动作是否是业务语言中的动作。
- 创建时是否需要校验不变量。
- 创建时是否要组装内部实体和值对象。
- 调用方是否不应该知道内部结构。
- 创建过程是否可能变化。

如果只是 `return new X()`，工厂价值不大。

## 创建行为也是领域行为

很多业务规则发生在创建时：

- 创建订单时必须至少有一个订单项。
- 创建 Sprint 时日期范围不能超过规定长度。
- 创建退款单时必须引用可退款订单。
- 创建出库任务时必须有有效库存预占。

这些规则不应该散落在 Controller 或应用服务里。简单规则可以放静态工厂，复杂规则可以放领域工厂。

## 工厂命名要表达业务意图

| 不好的命名 | 更好的命名 |
|---|---|
| create(...) | createDraftOrder(...) |
| build(...) | planSprint(...) |
| generate(...) | createReturnRequest(...) |
| newInstance(...) | openAccount(...) |

如果创建动作在业务里有明确名称，用业务名称做方法名。

## 工厂和构造函数的配合

常见做法：

```text
构造函数 private/protected
静态工厂表达简单创建语义
领域工厂处理复杂创建流程
```

这样可以避免调用方绕过创建规则。

## 什么时候需要工厂

适合使用工厂的信号：

- 创建对象需要多个步骤。
- 创建时要校验业务规则。
- 构造函数参数很多且容易传错。
- 创建过程需要选择不同策略或子类型。
- 创建聚合时要创建内部实体和值对象。
- 调用方不应知道内部结构。

不一定需要工厂的场景：

- 对象很简单。
- 构造规则很少。
- 静态创建方法已经足够表达业务。

## 构造函数、静态工厂、领域工厂

| 方式 | 适合场景 |
|---|---|
| 构造函数 | 简单值对象或简单实体 |
| 静态工厂方法 | 创建语义明确，规则不复杂 |
| 领域工厂 | 创建过程复杂，涉及多个对象或策略 |

## 静态工厂示例

```java
public class Order {
    public static Order createDraft(OrderId id, CustomerId customerId) {
        return new Order(id, customerId, OrderStatus.DRAFT);
    }
}
```

相比直接构造：

```java
new Order(id, customerId, "DRAFT");
```

静态工厂能表达业务语义。

## 领域工厂示例

复杂创建：

```java
public interface OrderFactory {
    Order create(CreateOrderRequest request);
}
```

实现中可以：

- 校验租户状态
- 创建订单项
- 计算初始金额
- 选择价格策略
- 生成订单号
- 产生 `OrderCreated` 事件

注意：如果工厂需要访问外部系统或数据库，要注意依赖方向。可以由应用服务准备好必要数据，再调用领域工厂。

## 工厂和应用服务的分工

| 职责 | 应用服务 | 工厂 |
|---|---|---|
| 接收命令 | 是 | 否 |
| 权限校验 | 是 | 否 |
| 事务控制 | 是 | 否 |
| 创建领域对象 | 编排 | 执行创建规则 |
| 业务创建规则 | 不应散落 | 是 |
| 保存聚合 | 是 | 否 |

应用服务负责用例，工厂负责模型创建。

## 工厂和聚合

聚合创建时尤其适合使用工厂，因为聚合内部结构不应该暴露给外部。

例如创建 Sprint：

```java
public class SprintFactory {
    public Sprint plan(ProjectId projectId, SprintGoal goal, DateRange period) {
        if (period.durationInDays() > 30) {
            throw new IllegalArgumentException("sprint is too long");
        }
        return Sprint.plan(projectId, goal, period);
    }
}
```

这里工厂保护创建规则，聚合继续保护生命周期规则。

## 工厂不是反贫血万能药

不要把所有业务行为都塞进工厂。工厂只负责创建，不负责后续生命周期行为。

错误：

```java
orderFactory.pay(order);
orderFactory.cancel(order);
```

更合理：

```java
order.pay();
order.cancel();
```

## 常见误区

- 任何对象都建工厂，导致过度设计。
- 工厂只是在调用构造函数，没有业务价值。
- 工厂承担对象生命周期行为。
- 工厂直接依赖大量基础设施细节。
- 工厂变成创建 DTO 的工具类。

### 供应链工厂示例

创建出库单不是简单 `new ShipmentOrder()`。它可能需要：

- 已有有效库存预占。
- 订单已经支付。
- 收货地址完整。
- 仓库可履约。
- 出库单初始状态为 `CREATED`。
- 产生 `ShipmentOrderCreated` 事件。

可以建模为：

```java
public class ShipmentOrderFactory {
    public ShipmentOrder createFromReservation(
        OrderId orderId,
        InventoryReservation reservation,
        Address shippingAddress
    ) {
        if (!reservation.isConfirmed()) {
            throw new IllegalStateException("reservation is not confirmed");
        }
        return ShipmentOrder.create(orderId, reservation.id(), shippingAddress);
    }
}
```

### 什么时候不用工厂

不需要为了所有对象都建工厂。比如简单值对象：

```java
new Money(amount, currency)
```

或者：

```java
Money.of(amount, currency)
```

就足够了。工厂的价值在于表达复杂创建规则，不是制造一层空壳。
