# 第05章：实体

> 本章目标：理解实体不是数据库表的映射，而是有稳定身份、生命周期和业务行为的领域对象。

## 本章在全书中的位置

第5章开始进入战术设计。实体是最容易被误用的模式，因为 Java 开发者常把数据库表、ORM Entity、DTO 和 DDD Entity 混在一起。

## 一句话理解

实体的核心是身份和生命周期，而不是字段。

## 本章精读路线

读实体这一章时，把注意力从“这个类有哪些字段”转到“这个对象如何保持自己是自己”。

你需要重点理解三件事：

1. 身份：什么让它区别于另一个对象。
2. 生命周期：它会经历哪些状态。
3. 行为：它能执行哪些业务动作来改变自己。

如果一个类没有身份、没有生命周期、没有行为，它很可能不是 DDD 意义上的实体，只是数据结构。

## 实体生命周期建模

实体经常和状态机有关。以订单为例：

| 状态 | 可执行动作 | 不可执行动作 |
|---|---|---|
| DRAFT | addItem、submit、cancel | pay、ship |
| SUBMITTED | pay、cancel | addItem、ship |
| PAID | ship、refund | addItem、pay |
| SHIPPED | complete、refund | cancel、addItem |
| COMPLETED | refund | pay、ship、cancel |
| CANCELLED | 无 | submit、pay、ship |

状态机不一定要用框架实现，但这些规则必须在模型中有清晰位置。最糟糕的是散落在多个 service 的 if/else 里。

## 身份设计的深入判断

| 选择 | 优点 | 风险 |
|---|---|---|
| 数据库自增ID | 简单，查询方便 | 创建前没有身份，跨系统不稳定 |
| UUID | 创建时可生成，分布式友好 | 可读性差 |
| 业务单号 | 可读，可对外沟通 | 规则变化会影响身份设计 |
| 外部系统ID | 对接方便 | 外部变化会污染本地模型 |

核心域中，建议把身份建成值对象，而不是到处使用 `Long` 或 `String`。

## 实体行为命名

实体方法名应该像业务动作：

| 不好的方法名 | 更好的业务方法 |
|---|---|
| setStatus(PAID) | markPaid() |
| updateAddress(address) | changeShippingAddress(address) |
| setAssignee(userId) | assignTo(userId) |
| setClosed(true) | close() |
| setDeleted(true) | archive() / cancel() |

业务方法名本身就是通用语言的一部分。

## 什么是实体

实体表示一个业务对象，即使它的属性发生变化，业务上仍然认为它是同一个对象。

| 对象 | 为什么是实体 |
|---|---|
| 用户 | 昵称、手机号变化后仍是同一个用户 |
| 订单 | 状态、金额、地址变化后仍是同一张订单 |
| 项目任务 | 标题、负责人、状态变化后仍是同一项任务 |
| 合同 | 条款修订后仍对应同一份合同 |

判断句式：

```text
如果这个对象的属性都变了，它还是不是同一个业务对象？
```

如果答案是“是”，它可能是实体。

## 身份标识

实体需要唯一身份，但身份不一定等于数据库自增 ID。

| 身份来源 | 例子 | 注意点 |
|---|---|---|
| 用户提供 | 手机号、邮箱、证件号 | 可能变更或不稳定 |
| 应用生成 | UUID、雪花 ID、业务单号 | 适合创建时就需要身份 |
| 持久化生成 | 数据库自增 ID | 创建前没有身份，不利于领域事件 |
| 外部上下文提供 | 第三方订单号 | 需要防腐层翻译和保护 |

DDD 中更推荐使用明确的身份值对象：

```java
public record OrderId(String value) {
    public OrderId {
        if (value == null || value.isBlank()) {
            throw new IllegalArgumentException("order id is required");
        }
    }
}
```

这样比直接传 `String` 或 `Long` 更能表达业务语义。

## 实体应该有行为

贫血实体：

```java
public class Order {
    private String status;
    public void setStatus(String status) {
        this.status = status;
    }
}
```

问题是：任何人都可以把订单改成任何状态，业务规则失控。

更好的实体：

```java
public class Order {
    private OrderId id;
    private OrderStatus status;

    public void submit() {
        if (status != OrderStatus.DRAFT) {
            throw new IllegalStateException("only draft order can be submitted");
        }
        this.status = OrderStatus.SUBMITTED;
    }

    public void cancel() {
        if (status == OrderStatus.PAID) {
            throw new IllegalStateException("paid order cannot be cancelled directly");
        }
        this.status = OrderStatus.CANCELLED;
    }
}
```

这里的关键不是方法变多，而是状态变化必须经过业务动作。

## 发现实体的本质特征

实体设计时，先找“什么让它成为它”，而不是先列字段。

以订单为例：

| 问题 | 回答 |
|---|---|
| 它的身份是什么？ | OrderId / 订单号 |
| 它的生命周期是什么？ | 草稿、已提交、已支付、已发货、已完成、已取消 |
| 它必须保护什么规则？ | 已支付不能随意改商品，取消要符合条件 |
| 它会产生什么事件？ | OrderSubmitted、OrderPaid、OrderCancelled |
| 它和哪些值对象协作？ | Money、Address、Quantity |

## 实体和数据库表的关系

数据库表是持久化结构，实体是业务模型。两者可以相似，但不应该强制一一对应。

常见情况：

- 一个实体可能对应多张表。
- 多个值对象可能嵌入一张表。
- 读模型可能完全不对应聚合结构。
- 数据库主键可能不是业务身份。

如果数据库结构决定了实体结构，领域模型就会被持久化细节牵着走。

## 验证和状态变化

实体内部应保护与自己生命周期相关的规则。

适合放在实体里的规则：

- 订单状态流转是否合法。
- 任务是否可以关闭。
- 合同是否可以提交审批。
- 用户是否可以变更绑定手机。

不适合放在实体里的规则：

- 发送短信。
- 调用支付网关。
- 查询数据库。
- 跨多个聚合的大流程编排。

## Java落地建议

实体类中可以避免暴露无脑 setter：

```java
public class Task {
    private TaskId id;
    private TaskStatus status;
    private UserId assigneeId;

    public void assignTo(UserId userId) {
        if (status == TaskStatus.CLOSED) {
            throw new IllegalStateException("closed task cannot be assigned");
        }
        this.assigneeId = userId;
    }

    public void close() {
        if (assigneeId == null) {
            throw new IllegalStateException("unassigned task cannot be closed");
        }
        this.status = TaskStatus.CLOSED;
    }
}
```

领域对象可以有少量 getter，但状态修改尽量通过业务方法完成。

## 常见误区

- 每张表都生成一个实体。
- 实体只有字段，没有行为。
- 用 setter 绕过业务规则。
- 把 DTO 当实体。
- 把 JPA/MyBatis 的持久化对象和领域实体完全混同。
- 所有校验都放在应用服务里。

### 供应链实体候选示例

| 实体 | 身份 | 生命周期 | 关键行为 |
|---|---|---|---|
| Order | OrderId | 草稿、已提交、已支付、已取消 | submit、markPaid、cancel |
| InventoryReservation | ReservationId | 已创建、已确认、已释放、已扣减 | confirm、release、deduct |
| ShipmentOrder | ShipmentOrderId | 已创建、拣货中、已复核、已发货 | startPicking、confirm、ship |
| ReturnOrder | ReturnOrderId | 已申请、已审核、已入库、已退款 | approve、receive、refund |

这些对象不是因为有表才是实体，而是因为业务会追踪它们的身份和生命周期。

### 从贫血实体到领域实体

贫血模型：

```java
order.setStatus("PAID");
orderMapper.update(order);
```

领域模型：

```java
order.markPaid(paymentId, paidAmount);
orderRepository.save(order);
```

差异在于：`markPaid` 可以检查订单是否已提交、支付金额是否匹配、是否重复支付，并产生 `OrderPaid` 事件。
