# 附录A：聚合与事件源

> 本章目标：理解事件源如何用事件序列保存聚合状态，以及它的价值和成本。

## 本章在全书中的位置

附录A是进阶内容。它建立在聚合、领域事件、最终一致性之上。初学者应先掌握第8章领域事件和第10章聚合，再学习事件源。

## 一句话理解

事件源不是保存当前状态，而是保存导致当前状态的一系列领域事件。

## 本章精读路线

事件源是进阶内容，读的时候不要被“高级架构”吸引。先问两个问题：

1. 业务是否真的关心完整历史？
2. 事件是否能自然表达业务事实？

如果只是为了审计字段变化，普通操作日志可能够用。如果业务核心就是一连串事实的演进，事件源才更自然。

## 事件源和操作日志的区别

| 对比 | 操作日志 | 事件源 |
|---|---|---|
| 目的 | 记录谁做了什么 | 用事件作为事实来源 |
| 是否重建状态 | 通常不能 | 可以 |
| 事件语义 | 可能偏技术 | 必须是业务事实 |
| 模型关系 | 附属记录 | 核心存储模型 |
| 查询方式 | 查日志 | 通过投影构建读模型 |

不要把“保存一条日志”误认为事件源。

## 事件源的实现流程

```text
Command
→ Aggregate 处理命令
→ 产生 Domain Events
→ Event Store 追加事件
→ Projection 更新读模型
→ Subscriber 响应事件
```

重建聚合：

```text
读取某聚合的所有事件
→ 按 sequence 排序
→ 逐条 apply
→ 得到当前状态
```

这里的关键是事件必须足以表达状态变化。

## 采用事件源前的成本清单

| 成本 | 说明 |
|---|---|
| 事件建模成本 | 事件必须稳定、清晰、可演进 |
| 查询成本 | 需要投影读模型 |
| 版本成本 | 旧事件长期存在，需要兼容 |
| 运维成本 | Event Store、重放、修复、监控 |
| 调试成本 | 当前状态来自事件序列，需要追踪 |
| 团队成本 | 团队要理解事件建模和最终一致 |

如果团队还没掌握聚合和领域事件，不建议直接上事件源。

## 普通持久化与事件源

普通持久化保存当前状态：

```text
Order(id=O1, status=PAID, totalAmount=100)
```

事件源保存历史事实：

```text
OrderCreated
OrderItemAdded
OrderSubmitted
PaymentCompleted
OrderPaid
```

通过重放事件，可以还原聚合当前状态。

## 事件源的价值

| 价值 | 说明 |
|---|---|
| 完整历史 | 每次变化都有业务事件记录 |
| 审计友好 | 可以追溯状态为什么变成这样 |
| 回放能力 | 可以用历史事件重建状态或读模型 |
| 业务表达自然 | 某些领域更关心发生过什么 |
| 支持事件驱动 | 事件天然成为集成和读模型更新来源 |

## 事件源适合什么场景

适合：

- 金融账户流水
- 订单生命周期审计
- 库存流水
- 工作流状态流转
- 协作编辑历史
- 风控决策记录

不适合：

- 简单 CRUD
- 历史不重要
- 团队缺少事件建模经验
- 查询需求简单但实时复杂
- 无法承担运维和版本演进成本

## 聚合如何应用事件

事件源聚合通常有两类方法：

```text
命令方法：执行业务规则，产生事件
应用事件方法：根据事件改变内部状态
```

示例：

```java
public class Order {
    private OrderId id;
    private OrderStatus status;
    private final List<DomainEvent> changes = new ArrayList<>();

    public void submit() {
        if (status != OrderStatus.DRAFT) {
            throw new IllegalStateException("only draft order can be submitted");
        }
        recordThat(new OrderSubmitted(id, Instant.now()));
    }

    private void recordThat(DomainEvent event) {
        apply(event);
        changes.add(event);
    }

    private void apply(DomainEvent event) {
        if (event instanceof OrderSubmitted) {
            this.status = OrderStatus.SUBMITTED;
        }
    }
}
```

重建聚合时，只 apply 历史事件，不再产生新事件。

## 事件存储

事件存储通常保存：

| 字段 | 作用 |
|---|---|
| aggregate_id | 聚合 ID |
| aggregate_type | 聚合类型 |
| event_id | 事件 ID |
| event_type | 事件类型 |
| event_version | 事件版本 |
| sequence | 聚合内事件序号 |
| payload | 事件内容 |
| occurred_at | 发生时间 |
| metadata | traceId、operatorId 等 |

事件序号可用于乐观并发控制。

## 快照

如果聚合事件很多，每次重放成本高，可以定期保存快照：

```text
Snapshot at version 100
→ replay events 101-120
```

快照是优化手段，不是事件源的核心。

## 事件版本演进

事件源最大挑战之一是事件会长期保存。事件结构变化时要考虑：

- 新旧版本兼容。
- upcaster 转换旧事件。
- 消费者兼容。
- 字段只能增加还是可以删除。
- 业务语义变化如何处理。

事件一旦发布，就像对外 API 一样需要治理。

## 查询模型

事件源通常配合 CQRS。写模型通过事件保存，读模型通过事件投影生成：

```text
Event Store
→ Projection
→ OrderDetailView / OrderListView / ReportView
```

不要直接让页面每次都重放事件查询。

## 常见误区

- 因为事件源高级就使用。
- 把技术日志当事件源。
- 事件不是业务事实，而是字段变化记录。
- 没有事件版本治理。
- 没有投影和读模型设计。
- 忽略幂等、顺序和并发。
- 聚合边界没想清楚就上事件源。

### 供应链事件源候选

适合考虑事件源的场景：

| 场景 | 为什么适合 |
|---|---|
| 库存流水 | 需要追踪每次预占、释放、扣减、调整 |
| 账户余额 | 当前余额必须能追溯来源 |
| 售后流程 | 审核、收货、退款、补偿需要完整过程 |

不一定适合：

| 场景 | 原因 |
|---|---|
| 商品基础资料 | 当前状态通常足够 |
| 普通后台配置 | 历史过程价值低 |
| 简单通知记录 | 日志或状态表更简单 |

### 库存事件源示例

```text
InventoryInitialized
InventoryReserved
InventoryReservationReleased
InventoryDeducted
InventoryAdjusted
```

当前可用库存不是直接 update 得到，而是由这些事件重放或投影得出。

### 最终判断

事件源不是 DDD 的必选项。它适合“历史事实本身就是业务核心”的场景。如果只是为了记录日志，用普通审计日志更简单。
