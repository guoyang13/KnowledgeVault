# 第01章：DDD入门

> 本章目标：先判断为什么需要 DDD，再理解 DDD 不是代码分层技巧，而是一套围绕复杂业务建立模型、语言和边界的方法。

## 本章在全书中的位置

第1章是整本书的入口。它不急着讲实体、值对象、聚合，而是先回答三个问题：

1. 什么样的软件问题值得使用 DDD？
2. DDD 相比普通 CRUD、三层架构、数据表驱动开发解决了什么？
3. 为什么只学战术模式会变成 DDD-Lite？

如果这一章没有想清楚，后面很容易把 DDD 学成“把代码放进 domain 包”。

## 初学者先抓住一句话

DDD 是在业务复杂度很高时，让业务专家和开发团队围绕同一套语言、边界和模型协作，持续交付正确软件的方法。

## 本章精读路线

这一章要读三遍，每一遍关注点不同：

| 阅读轮次 | 关注点 | 读完要产出 |
|---|---|---|
| 第1遍 | DDD 为什么存在 | 写出“我的项目复杂在哪里” |
| 第2遍 | DDD 反对哪些常见做法 | 找出一个数据表驱动或贫血模型例子 |
| 第3遍 | DDD 如何开始落地 | 列出业务词表、领域专家、核心流程 |

本章不要急着背模式名。它最重要的作用是让你建立判断：什么时候 DDD 值得投入，什么时候只是普通 CRUD。

## 复杂度识别清单

判断一个系统是否适合 DDD，可以从下面几类复杂度入手：

| 复杂度来源 | 低复杂度表现 | 高复杂度表现 |
|---|---|---|
| 业务规则 | 简单增删改查 | 状态流转多、例外多、审批多 |
| 语言一致性 | 业务和技术词基本一致 | 同一个词在不同团队中含义不同 |
| 变化频率 | 需求很少变化 | 规则频繁调整，且影响多个模块 |
| 协作关系 | 单团队维护 | 多团队、多系统、多上下游 |
| 数据含义 | 字段含义稳定 | 字段在不同流程中含义不同 |
| 流程跨度 | 单个事务完成 | 跨系统、跨组织、长流程 |

如果你的系统在 3 项以上偏向高复杂度，就值得认真使用 DDD。

## 从普通三层到DDD的转变

普通三层常见思路：

```text
Controller 接请求
→ Service 写流程和判断
→ Mapper 操作表
→ Entity 承载字段
```

DDD 的思路：

```text
先识别领域和上下文
→ 用通用语言命名模型
→ 在聚合中保护业务规则
→ 应用服务编排用例
→ 基础设施负责技术细节
```

二者最大的差异不是包名，而是业务规则放在哪里。

## 初学者落地的最小动作

不用一开始就重构整个系统。先做 5 个最小动作：

1. 建立一个业务词表，记录词的定义、使用场景和歧义。
2. 找一个复杂流程，画出参与角色、系统和状态变化。
3. 找一个大 service，标出哪些逻辑属于业务规则。
4. 找一个字段很多的实体，判断哪些字段其实属于不同上下文。
5. 找一个状态流转，改成有业务方法的模型对象。

这 5 个动作比空谈“DDD架构”更有效。

## 核心概念

| 概念 | 解释 | 初学者判断 |
|---|---|---|
| 领域 | 软件要解决的业务问题空间 | 这个系统到底在帮业务完成什么？ |
| 领域模型 | 对业务概念、规则、行为的抽象 | 它是不是只表达数据库字段？ |
| 通用语言 | 业务和研发共同使用的模型语言 | 会议、文档、代码是否同词同义？ |
| 战略设计 | 划分边界、子域和上下文关系 | 有没有先问业务边界？ |
| 战术设计 | 在边界内设计实体、值对象、聚合等 | 模型对象是否承担业务规则？ |
| DDD-Lite | 只用了 DDD 名词，没做边界和语言建设 | 是否只有 Entity/Service/Repository？ |

## 为什么需要 DDD

DDD 主要处理业务复杂度，而不是技术复杂度。

技术复杂度通常来自：

- 框架选型
- 高并发
- 分布式事务
- 缓存一致性
- 数据库性能

业务复杂度通常来自：

- 业务规则多且经常变化
- 同一个词在不同部门含义不同
- 例外场景很多
- 人工流程和系统流程交织
- 模型一改牵动多个团队

DDD 的价值在于，它先把业务复杂度变成可讨论、可命名、可边界化、可测试的模型。

## DDD反对什么

### 1. 数据表驱动一切

常见做法：

```text
先设计表
→ 生成 Entity
→ 写 Mapper
→ 写 Service
→ Controller 调 Service
```

这种方式适合简单 CRUD，但在复杂业务中容易让代码变成“数据库操作脚本”。业务规则散落在 service、mapper、前端和 SQL 里，模型没有中心。

DDD 不是不需要数据库，而是要求先理解业务模型，再选择持久化方式。

### 2. 贫血模型

贫血模型的特征：

- 对象只有字段和 getter/setter。
- 所有判断逻辑都写在 service。
- 实体不能保护自己的状态。
- 单元测试必须依赖 service 才能验证业务规则。

DDD 希望核心业务行为回到模型对象中。

### 3. 技术分层冒充业务建模

把代码分成 controller、service、mapper 并不等于 DDD。这只是技术分层。

DDD 更关心：

- 业务语言是否统一？
- 边界是否清楚？
- 聚合是否保护不变量？
- 上下文之间是否正确集成？

## DDD带来的业务价值

| 价值 | 具体含义 |
|---|---|
| 更准确的业务理解 | 开发和业务不再各说各话 |
| 更清晰的模型边界 | 不同业务语义不混在一个大模型里 |
| 更好的协作 | 领域专家可以参与设计讨论 |
| 更强的演进能力 | 变化被限制在合理边界内 |
| 更可靠的规则实现 | 核心规则集中在模型中，可测试 |
| 更好的架构治理 | 系统集成关系更清楚 |

## 什么时候不必重度DDD

DDD 不是银弹。下面场景不需要重度投入：

- 纯 CRUD 后台
- 业务规则极少
- 生命周期很短的临时系统
- 团队没有复杂协作问题
- 核心价值不在业务模型，而在算法、数据采集或展示

但即使不做完整 DDD，也可以借用通用语言、值对象、边界意识等思想。

## 本章练习

选择你熟悉的一个系统，回答：

1. 它最核心的业务目标是什么？
2. 业务人员最常说的 10 个词是什么？
3. 哪些词在不同团队中含义不一致？
4. 哪个 service 最大、最难改？
5. 这个 service 中哪些逻辑其实属于业务模型？

## 阅读检查

- 我能区分业务复杂度和技术复杂度吗？

- 我能解释为什么 DDD 不是目录结构吗？

## 深化精读补充

### 逐节精读抓手

读本章时，可以把内容拆成 4 个问题：

| 阅读块 | 要抓住的问题 | 不要误读成 |
|---|---|---|
| 我能DDD吗 | 我的业务复杂度是否值得建模投入 | 所有项目都必须DDD |
| 为什么需要DDD | 复杂业务为什么不能只靠数据库和service | DDD只是代码风格 |
| 如何DDD | 如何从语言、边界、模型开始 | 先搭框架、先分包 |
| 业务价值和挑战 | DDD带来什么收益，也会付出什么成本 | DDD没有代价 |

### 供应链业务例子

假设你在做供应链系统，业务提出“订单发货前要校验库存”。普通 CRUD 视角会问：

```text
查哪张订单表？
查哪张库存表？
怎么 update 库存数量？
```

DDD 视角会先问：

```text
这里的订单是交易订单、履约订单，还是仓储出库任务？
库存是可售库存、实物库存、预占库存，还是在途库存？
校验失败是阻止支付、阻止出库，还是进入缺货补偿流程？
谁对库存预占这条规则负责？
```

这就是本章最重要的转变：先弄清业务语言和业务规则，再谈表和接口。
