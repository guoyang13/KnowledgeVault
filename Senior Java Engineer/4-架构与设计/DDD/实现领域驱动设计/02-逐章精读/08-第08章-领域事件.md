# 第08章：领域事件

> 本章目标：学会用事件表达业务上已经发生的重要事实，并用它支持模型解耦、审计和最终一致性。

## 本章在全书中的位置

第8章是战术设计的关键章节。领域事件不是技术消息的另一个名字，它首先是业务事实。

## 一句话理解

领域事件表示领域中已经发生、业务上值得关注的事实。

## 本章精读路线

读领域事件时要先忘掉 MQ。先把事件当成业务语言中的事实句子：

```text
订单已提交
支付已完成
库存已预占
Sprint已关闭
```

等你能清楚表达业务事实后，再考虑它是否要通过 MQ、应用事件、事务消息或本地消息表发布。

## 事件设计的四个层次

| 层次 | 问题 | 示例 |
|---|---|---|
| 业务事实 | 发生了什么 | OrderPaid |
| 模型来源 | 哪个聚合产生 | Order 聚合 |
| 消费意图 | 谁关心这件事 | 库存、仓储、通知 |
| 技术投递 | 怎么送出去 | 本地事件、MQ、Outbox |

不要反过来从“我要发一条 MQ”开始设计事件。

## 事件粒度

事件太粗会让消费者无法理解，事件太细会造成噪声。

| 粒度 | 问题 |
|---|---|
| `OrderUpdated` | 太粗，不知道业务上发生了什么 |
| `OrderStatusChangedToPaid` | 暴露内部状态实现 |
| `OrderPaid` | 清晰表达业务事实 |
| `OrderAmountFieldChanged` | 太技术化 |

优先使用业务人员能理解的过去式名称。

## 领域事件到集成事件

内部领域事件不一定直接对外发布。可以转换：

```text
Domain Event: OrderPaid
→ Integration Event: TradeOrderPaidV1
→ MQ Message
```

这样可以避免内部模型变化直接影响外部消费者。

## 命令和事件

| 类型 | 含义 | 命名方式 | 是否一定成功 |
|---|---|---|---|
| 命令 | 希望系统做某事 | 动词开头 | 可能失败 |
| 事件 | 某事已经发生 | 过去式 | 已经发生 |

例子：

| 命令 | 事件 |
|---|---|
| SubmitOrder | OrderSubmitted |
| PayOrder | OrderPaid |
| ReserveInventory | InventoryReserved |
| ShipOrder | OrderShipped |

命令是请求，事件是事实。

## 领域事件应该来自业务语言

好的事件：

- `OrderSubmitted`
- `PaymentCompleted`
- `InventoryReserved`
- `SprintClosed`

不好的事件：

- `OrderTableUpdated`
- `SendMqMessage`
- `StatusChanged`
- `SyncDataFinished`

前者表达业务事实，后者表达技术动作或模糊变化。

## 领域事件包含什么

领域事件通常包含：

| 字段 | 作用 |
|---|---|
| eventId | 事件唯一标识，支持幂等 |
| occurredAt | 发生时间 |
| aggregateId | 事件来源聚合 |
| business data | 消费方需要的稳定业务数据 |
| traceId | 链路追踪 |
| version | 事件版本 |

不要把整个聚合对象塞进事件。事件是发布语言，应稳定、明确、少泄漏内部细节。

## 领域事件的用途

| 用途 | 例子 |
|---|---|
| 聚合内记录事实 | 订单提交后记录 OrderSubmitted |
| 模型内解耦 | 订单支付后触发积分、通知 |
| 跨上下文集成 | 支付上下文通知仓储上下文 |
| 审计 | 记录关键业务变化 |
| 最终一致性 | 一个聚合变化后，另一个聚合异步响应 |
| 更新读模型 | 事件驱动报表、列表、搜索索引 |

## 聚合发布事件

示例：

```java
public class Order {
    private final List<DomainEvent> events = new ArrayList<>();

    public void submit() {
        if (status != OrderStatus.DRAFT) {
            throw new IllegalStateException("only draft order can be submitted");
        }
        this.status = OrderStatus.SUBMITTED;
        this.events.add(new OrderSubmitted(id, Instant.now()));
    }

    public List<DomainEvent> pullEvents() {
        List<DomainEvent> pulled = List.copyOf(events);
        events.clear();
        return pulled;
    }
}
```

应用服务负责发布：

```java
order.submit();
orderRepository.save(order);
eventPublisher.publish(order.pullEvents());
```

这样领域模型负责产生事实，基础设施负责投递事实。

## 同步事件和异步消息

领域事件不一定马上等于 MQ 消息。

| 层次 | 含义 |
|---|---|
| 领域事件 | 领域模型中发生的业务事实 |
| 应用内事件 | 进程内解耦处理 |
| 集成事件 | 对外发布给其他上下文 |
| MQ消息 | 技术投递形式 |

可以把领域事件转换成集成事件，再通过 MQ 发出。不要让 MQ 消息格式反向控制领域事件设计。

## 最终一致性

领域事件常用于跨聚合、跨上下文的最终一致。

示例：

```text
OrderPaid
→ Inventory reserves stock
→ InventoryReserved
→ Warehouse creates shipment task
```

这里不需要一个大事务覆盖订单、库存、仓储。每个上下文处理自己的规则，通过事件协调。

## 幂等和可靠性

事件落地时要考虑：

- 事件 ID 幂等
- 发送失败重试
- 消费失败重试
- 事件顺序
- 死信队列
- 事件版本兼容
- 本地消息表或事务消息

这些属于工程实现，但设计事件时要提前留出空间。

## 常见误区

- 把技术消息当领域事件。
- 事件命名不是过去式。
- 事件过大，包含内部对象全量快照。
- 所有状态变化都发事件，造成噪声。
- 用事件掩盖错误边界。
- 只关注 MQ，不关注业务事实。

## 本章练习

为一个订单流程设计事件：

| 命令 | 可能产生的事件 | 消费方 | 是否集成事件 |
|---|---|---|---|
| SubmitOrder | OrderSubmitted | 库存、通知 | 是 |
| PayOrder | OrderPaid | 仓储、积分 | 是 |

再检查每个事件：

- 是否是过去式？
- 是否表达业务事实？
- 是否包含 eventId 和 occurredAt？
- 是否避免泄漏内部模型？

## 阅读检查

- 我能区分命令和事件吗？
- 我知道领域事件和 MQ 消息不是一回事吗？
- 我能判断哪些事件值得对外发布吗？
- 我知道事件最终一致需要幂等和重试吗？

### 供应链事件流示例

```text
OrderPaid
→ InventoryReservationRequested
→ InventoryReserved
→ ShipmentOrderCreated
→ ShipmentPicked
→ ShipmentCompleted
```

这里每个事件都是已经发生的事实，而不是命令。`ReserveInventory` 是命令，`InventoryReserved` 才是事件。

### 事件字段设计示例

`OrderPaid` 可以包含：

```text
eventId
occurredAt
orderId
paymentId
paidAmount
currency
tenantId
traceId
version
```

不要包含整个订单聚合。消费者通常不需要订单所有字段，只需要完成自己职责所需的信息。

### 事件设计产出模板

```text
消费者：
是否对外发布：
幂等键：
失败重试策略：
版本策略：
```

### 事件命名练习

| 不推荐 | 推荐 | 原因 |
|---|---|---|
| UpdateOrderStatus | OrderPaid | 前者是技术动作，后者是业务事实 |
| SyncInventory | InventoryReserved | 前者模糊，后者明确 |
| SendShipmentMessage | ShipmentCreated | 不把消息发送当业务事件 |

### 判断是否进入下一章

进入模块章节前，先确认你能把事件放回正确上下文：事件名来自业务语言，事件字段不泄漏内部模型，消费者不直接依赖发布方聚合。
