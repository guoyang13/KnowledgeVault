# 第07章：领域服务

> 本章目标：学会判断哪些业务行为不适合放入实体和值对象，而应该建模为领域服务。

## 本章在全书中的位置

第5章和第6章分别讲实体和值对象。第7章补上一个重要问题：有些业务行为确实属于领域，但不自然属于任何一个实体或值对象，这时可以使用领域服务。

## 一句话理解

领域服务是无状态的领域行为，适合承载不自然归属某个实体或值对象的业务操作。

## 本章精读路线

读领域服务时，最重要的是克制。很多初学者一看到 service 就以为所有业务逻辑都能放进去，这正好回到了传统贫血模型。

读本章时按下面顺序判断：

```text
这个行为能否放到实体？
→ 不能，能否放到值对象？
→ 不能，是否真的是领域规则？
→ 是，再考虑领域服务
```

领域服务是兜底选择，不是默认选择。

## 领域服务的三种典型场景

| 场景 | 说明 | 例子 |
|---|---|---|
| 跨多个领域对象的计算 | 规则不属于单个对象 | 定价、库存分配 |
| 策略型领域规则 | 多种业务策略可替换 | 最近仓优先、成本最低优先 |
| 转换型领域概念 | 一个领域概念转换成另一个领域概念 | 汇率换算、计量单位转换 |

不是所有跨对象操作都是领域服务。如果只是流程编排，属于应用服务；如果只是技术调用，属于基础设施服务。

## 领域服务命名

领域服务命名要体现业务能力：

| 不好的命名 | 更好的命名 |
|---|---|
| OrderDomainService | PricingService / OrderEligibilityService |
| CommonService | InventoryAllocationService |
| DataConvertService | CurrencyExchangeService |
| CheckService | RiskAssessmentService |

如果名字只能叫 `XxxDomainService`，说明你可能还没有找准领域概念。

## 领域服务的依赖控制

领域服务可以依赖领域对象和领域层定义的接口，但要小心基础设施依赖。

可以：

```java
public interface ExchangeRateProvider {
    ExchangeRate getRate(Currency from, Currency to);
}
```

不建议领域服务直接：

```java
RestTemplate.getForObject(...)
mapper.selectById(...)
redisTemplate.opsForValue()
```

这些技术细节应由基础设施实现。

## 领域服务不是什么

在很多 Java 项目里，`XxxService` 是万能层。但 DDD 中的领域服务很克制。

它不是：

- Controller 的后置处理器
- DAO 的包装
- 事务脚本
- HTTP Client
- 工具类
- 应用用例编排器

这些通常分别属于接口层、基础设施层、应用层或通用工具。

## 什么时候需要领域服务

判断问题：

1. 这个操作是否是业务语言中的动作？
2. 它是否不自然属于某个实体或值对象？
3. 它是否涉及多个领域对象或聚合？
4. 它是否需要表达一个独立领域概念？
5. 它是否通常无状态？

如果答案多为“是”，可以考虑领域服务。

## 领域服务例子

### 定价服务

定价可能涉及商品、客户等级、活动、优惠券、区域、时间。它不一定自然属于订单，也不属于商品。

```java
public interface PricingService {
    Money calculatePrice(PricingContext context);
}
```

### 库存分配服务

库存分配可能涉及多个仓、批次、策略。

```java
public interface InventoryAllocationService {
    AllocationPlan allocate(AllocationRequest request);
}
```

### 风控判定服务

风控判定可能涉及用户、订单、历史行为、策略。

```java
public interface RiskAssessmentService {
    RiskDecision assess(RiskContext context);
}
```

## 应用服务、领域服务、基础设施服务对比

| 类型 | 主要职责 | 是否表达业务规则 | 例子 |
|---|---|---|---|
| 应用服务 | 编排用例、事务、权限、事件发布 | 不应承载核心规则 | SubmitOrderApplicationService |
| 领域服务 | 表达领域行为和规则 | 是 | PricingService |
| 基础设施服务 | 技术能力实现 | 否 | SmsClient、PaymentGatewayClient |

## 错误示例

应用服务变胖：

```java
public void submitOrder(SubmitOrderCommand command) {
    Order order = orderRepository.findById(command.orderId()).orElseThrow();
    if (order.getStatus() != DRAFT) {
        throw new IllegalStateException();
    }
    if (order.getItems().isEmpty()) {
        throw new IllegalStateException();
    }
    BigDecimal total = calculateTotal(order);
    if (total.compareTo(BigDecimal.ZERO) <= 0) {
        throw new IllegalStateException();
    }
    order.setStatus(SUBMITTED);
    orderRepository.save(order);
}
```

问题是应用服务承载了太多规则。

更好的方向：

```java
public void submitOrder(SubmitOrderCommand command) {
    Order order = orderRepository.findById(command.orderId()).orElseThrow();
    order.submit(pricingService);
    orderRepository.save(order);
    eventPublisher.publish(order.pullEvents());
}
```

具体是否让实体依赖领域服务，要看模型设计。也可以由应用服务先调用领域服务，再把结果传给聚合。

## 领域服务接口放哪里

如果领域服务表达纯业务能力，接口可放领域层：

```text
domain
  service
    PricingService
```

如果实现需要数据库、外部接口或规则引擎，实现放基础设施层：

```text
infrastructure
  pricing
    RuleEnginePricingService
```

## 领域服务和策略模式

很多领域服务背后是策略选择。

```java
public interface AllocationPolicy {
    AllocationPlan allocate(AllocationRequest request);
}
```

不同策略：

- 最近仓优先
- 成本最低优先
- 库龄优先
- 指定仓优先

领域服务可以协调策略，但策略本身也应该使用业务语言命名。

## 常见误区

- 一遇到逻辑就建领域服务，导致实体和值对象贫血。
- 把应用服务命名成领域服务。
- 领域服务持有大量状态。
- 领域服务直接处理 HTTP、SQL、MQ 细节。
- 领域服务方法名是技术动作，不是业务语言。

### 供应链领域服务示例

库存分配是典型领域服务候选，因为它可能涉及多个仓库、多个批次、多种策略：

```text
输入：SKU、数量、收货地址、渠道、优先级
规则：最近仓优先、成本最低、先进先出、指定仓优先
输出：分配计划 AllocationPlan
```

它不自然属于某一个 `Warehouse`，也不完全属于某一个 `Inventory`，因此可以建模为：

```java
public interface InventoryAllocationService {
    AllocationPlan allocate(AllocationRequest request);
}
```

### 判断一个Service的归属

| 方法 | 更可能属于 | 判断理由 |
|---|---|---|
| submitOrder | 应用服务 | 编排一个用例 |
| calculatePrice | 领域服务/值对象 | 价格规则属于业务 |
| callPaymentGateway | 基础设施服务 | 技术集成 |
| allocateInventory | 领域服务 | 跨库存对象的业务策略 |
| sendSms | 基础设施服务 | 技术能力 |
