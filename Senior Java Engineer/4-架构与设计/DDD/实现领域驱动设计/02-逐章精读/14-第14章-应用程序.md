# 第14章：应用程序

> 本章目标：理解应用层如何编排用例，让接口、DTO、事务、基础设施围绕领域模型工作。

## 本章在全书中的位置

第14章把前面的战略和战术模式串成可运行应用。到这里，领域模型已经不是孤立的类，而要被用户界面、API、消息、数据库和外部系统使用。

## 一句话理解

应用服务负责用例编排，领域模型负责业务规则，基础设施负责技术实现。

## 本章精读路线

读应用程序这一章时，要不断区分三种对象：

```text
DTO：适配接口
Command：表达用例意图
Domain Model：承载业务规则
```

很多项目的问题不是没有 service，而是 service 没有分清职责：Controller、Application Service、Domain Service、Infrastructure Service 混在一起。

## 应用服务的标准流程

一个命令型应用服务通常可以写成 8 步：

| 步骤 | 说明 |
|---|---|
| 1. 接收 Command | Command 表达用例意图 |
| 2. 权限/幂等检查 | 用例级控制，不是核心业务规则 |
| 3. 加载聚合 | 通过 Repository 获取聚合根 |
| 4. 调用领域行为 | 聚合或领域服务执行业务规则 |
| 5. 保存聚合 | Repository 持久化 |
| 6. 拉取领域事件 | 从聚合获得事件 |
| 7. 发布事件 | 应用层协调发布 |
| 8. 返回结果 | 返回 DTO 或简单结果 |

如果第4步里没有领域对象参与，而应用服务自己写完所有规则，模型就贫血了。

## 应用服务和事务

应用服务通常是事务边界，但事务边界不等于聚合边界。

推荐：

```text
一个应用服务方法
→ 一个本地事务
→ 修改一个聚合
→ 保存并发布事件
```

需要谨慎：

```text
一个应用服务方法
→ 同时修改订单、库存、仓储、支付多个聚合
```

这往往说明跨上下文流程应该改成事件或 Saga。

## DTO转换边界

典型转换路径：

```text
RequestDTO
→ Command
→ Domain Model
→ Result/ResponseDTO
```

每一层对象的变化原因不同：

| 对象 | 变化原因 |
|---|---|
| RequestDTO | API协议或前端页面变化 |
| Command | 用例意图变化 |
| Domain Model | 业务规则变化 |
| ResponseDTO | 展示或接口消费者变化 |

把它们混用会导致一个变化牵动所有层。

## 应用服务职责

应用服务通常负责：

- 接收命令。
- 做用例级权限、幂等、事务控制。
- 调用 Repository 加载聚合。
- 调用聚合或领域服务。
- 保存聚合。
- 发布领域事件。
- 返回 DTO。

应用服务不应该负责：

- 维护聚合内部不变量。
- 决定核心业务状态流转。
- 直接拼 SQL 处理业务规则。
- 承担复杂领域计算。

## 典型应用服务流程

```text
Controller / Consumer
→ Command
→ Application Service
→ Repository.find
→ Aggregate.method
→ Repository.save
→ EventPublisher.publish
→ DTO
```

Java 示例：

```java
public class OrderApplicationService {
    private final OrderRepository orderRepository;
    private final DomainEventPublisher eventPublisher;

    @Transactional
    public void submit(SubmitOrderCommand command) {
        Order order = orderRepository.findById(new OrderId(command.orderId()))
            .orElseThrow(() -> new IllegalArgumentException("order not found"));

        order.submit();
        orderRepository.save(order);
        eventPublisher.publish(order.pullEvents());
    }
}
```

应用服务薄，领域模型厚。

## DTO和领域对象

DTO 是边界对象，不是领域对象。

```text
Request DTO
→ Command
→ Domain Model
→ Response DTO
```

不要在领域层使用 Controller DTO，也不要把领域对象直接返回给前端。

原因：

- 前端字段变化不应影响领域模型。
- API 兼容性不应绑死内部对象。
- 领域对象可能包含不该暴露的信息。
- DTO 通常是视图导向，领域对象是规则导向。

## 命令对象

命令表达用户或系统希望执行的动作：

```java
public record SubmitOrderCommand(String orderId, String operatorId) {
}
```

命令不同于 DTO：

- DTO 适配接口。
- Command 适配用例。
- Domain Model 适配业务规则。

有时 DTO 和 Command 可以很像，但概念上不要混同。

## 查询和读模型

查询不一定要通过聚合。

命令侧：

```text
加载聚合 → 执行业务行为 → 保存聚合
```

查询侧：

```text
按页面/报表需要直接查询读模型 DTO
```

复杂列表、报表、搜索、详情页可以使用专门 QueryService，避免为了查询扩大聚合。

## 用户界面与领域模型

界面不是领域模型，但界面会影响用例设计。

要避免：

- 前端页面字段直接决定领域对象字段。
- 一个大保存按钮对应一个巨大事务。
- 页面 DTO 直接传进聚合。

更好的方式是把用户操作转成明确命令：

- `SubmitOrder`
- `ChangeShippingAddress`
- `CancelOrder`
- `AssignTask`

## 基础设施协作

应用服务可以使用基础设施能力，但应通过接口隔离：

```java
public interface DomainEventPublisher {
    void publish(List<DomainEvent> events);
}
```

具体实现：

```text
SpringEventPublisher
KafkaEventPublisher
RocketMqEventPublisher
```

应用层依赖抽象，基础设施层提供实现。

## 常见误区

- 应用服务写成大事务脚本。
- 业务规则全部在应用服务。
- Controller 直接调用 Repository。
- DTO 直接进入领域模型。
- 聚合直接返回给前端。
- 查询也强行走聚合，导致性能差。
- 应用层和基础设施层边界混乱。

### 供应链应用服务示例

订单支付回调应用服务：

```java
public class TradePaymentApplicationService {
    private final OrderRepository orderRepository;
    private final DomainEventPublisher eventPublisher;

    public void confirmPayment(ConfirmPaymentCommand command) {
        Order order = orderRepository.findById(command.orderId())
            .orElseThrow();

        order.markPaid(command.paymentId(), command.paidAmount());
        orderRepository.save(order);
        eventPublisher.publish(order.pullEvents());
    }
}
```

这个应用服务只编排：

- 加载订单。
- 调用领域行为。
- 保存订单。
- 发布事件。

它不应该自己判断订单能否支付，这个规则属于 `Order`。

### DTO到Command示例

```text
PaymentCallbackRequest
→ ConfirmPaymentCommand
→ Order.markPaid()
→ OrderPaid
→ PaymentCallbackResponse
```

前端/API字段变化只影响 Request/Response，业务用例变化影响 Command，领域规则变化影响聚合。
