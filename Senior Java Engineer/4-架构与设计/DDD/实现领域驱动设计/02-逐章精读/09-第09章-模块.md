# 第09章：模块

> 本章目标：学会按通用语言组织领域对象，而不是只按技术分层组织代码。

## 本章在全书中的位置

第9章看似只是讲包和命名空间，但它连接了战略设计和战术设计。模块是通用语言在代码结构中的体现。

## 一句话理解

模块是把内聚的领域概念放在一起，并用业务语言命名的代码组织单元。

## 本章精读路线

读模块这一章时，不要只理解成“Java包怎么放”。模块真正训练的是代码结构是否能反映业务语言。

重点看三件事：

1. 模块名称是否来自通用语言。
2. 模块内部对象是否围绕同一个业务概念内聚。
3. 模块之间是否存在不该有的依赖。

如果一个开发者打开包结构就能看出业务模型的大致结构，模块设计就是有价值的。

## 模块粒度判断

| 粒度 | 问题 | 建议 |
|---|---|---|
| 太粗 | `domain.model` 里放所有模型 | 按聚合或业务概念拆分 |
| 太细 | 每个类一个包 | 失去内聚，阅读成本高 |
| 技术化 | `entity`、`vo`、`service` | 改为业务概念包 |
| 跨上下文 | 一个包被多个上下文共用 | 拆到各自上下文，必要时用发布语言 |

模块不是越多越好，关键是它能否表达一个稳定业务概念。

## 模块依赖方向

模块之间要避免形成网状依赖：

```text
order → payment → inventory → order
```

如果出现循环依赖，通常说明：

- 聚合边界不清。
- 上下文边界不清。
- 某个模块承担了太多职责。
- 应该通过领域事件或应用服务协调，而不是模块互相调用。

## 模块命名示例

供应链上下文中，推荐用业务词：

```text
inventory.stock
inventory.reservation
inventory.allocation
warehouse.outbound
warehouse.receiving
aftersale.returnorder
```

不推荐：

```text
inventory.bean
inventory.manager
inventory.common
inventory.utils
```

技术词不能完全避免，但核心领域模块优先使用业务词。

## 为什么模块重要

如果代码只按技术分类：

```text
domain
  entity
  valueobject
  service
  repository
```

看起来整齐，但业务概念被拆散了。想理解订单，需要同时跳转 entity、valueobject、service、repository、event 多个包。

DDD 更关心业务内聚：

```text
domain
  model
    order
      Order
      OrderItem
      OrderId
      OrderRepository
      OrderSubmitted
    payment
      Payment
      PaymentId
      PaymentCompleted
```

这样阅读代码时，业务概念在一个局部范围内完整呈现。

## 模块命名原则

| 原则 | 说明 |
|---|---|
| 使用通用语言 | 包名应该是业务词，不是技术词 |
| 体现内聚 | 经常一起变化的模型对象放一起 |
| 控制依赖 | 模块之间避免随意互相访问内部对象 |
| 避免过早抽象 | 不要为了“通用”建立大而空的 common |
| 和限界上下文一致 | 模块边界不能跨越上下文语义 |

## 模块与限界上下文

限界上下文是更大的模型边界，模块是上下文内部的组织方式。

```text
trade context
  order module
  payment module
  promotion module

inventory context
  stock module
  reservation module
  allocation module
```

同名模块在不同上下文中可以有不同含义。比如 `order` 在交易上下文和仓储上下文不应强行共用。

## Java落地结构

以交易上下文为例：

```text
com.example.trade
  application
    order
      SubmitOrderCommand
      SubmitOrderHandler
  domain
    model
      order
        Order
        OrderId
        OrderItem
        OrderRepository
        OrderSubmitted
      payment
        Payment
        PaymentId
        PaymentCompleted
  infrastructure
    persistence
      order
      payment
```

这里的 `order`、`payment` 是业务模块，不是技术类型。

## common包的风险

很多项目会出现：

```text
common
  util
  enums
  dto
  model
```

风险：

- 所有上下文共享概念，边界被打穿。
- enum 被多个业务含义复用。
- DTO 成为跨上下文的事实模型。
- 任何修改影响面都很大。

可以有 common，但应限制为真正技术无关、业务无关或稳定的基础类型。领域概念不要轻易放 common。

## 模块依赖检查

每个模块都要问：

- 这个模块的核心业务词是什么？
- 它是否属于一个明确限界上下文？
- 它是否依赖了不该依赖的外部模块？
- 它暴露的是聚合根、领域服务接口还是内部细节？
- 包名是否能被业务专家理解？

## 常见误区

- 按 entity/service/repository 技术类型分包。
- 把所有枚举放到 common。
- 跨上下文共享模块。
- 包名使用缩写或技术词，业务看不懂。
- 模块只是目录，没有依赖约束。

## 本章练习

重构一个包结构设计：

| 当前包 | 问题 | 新模块名 | 调整理由 |
|---|---|---|---|
| entity | 业务概念混杂 | domain.model.order | 订单相关模型内聚 |

再检查：

- 是否按上下文组织？
- 是否按业务概念组织？
- 是否减少了 common？
- 是否减少了跨模块内部访问？

## 阅读检查

- 我能解释模块和限界上下文的区别吗？
- 我能按业务概念重新组织包吗？
- 我知道 common 包为什么危险吗？
- 我能识别技术分包带来的阅读问题吗？

## 深化精读补充

### 逐节精读抓手

| 阅读块 | 要抓住的问题 | 学习产出 |
|---|---|---|
| 模块为什么重要 | 包结构能否表达业务语言 | 当前包结构问题清单 |
| 模块命名 | 包名是否来自通用语言 | 新包名方案 |
| 模块依赖 | 是否存在循环依赖和跨边界访问 | 依赖关系图 |
| common风险 | 哪些共享其实污染模型 | common清理清单 |

### 供应链模块示例

不要这样组织：

```text
domain
  entity
  service
  repository
  event
```

推荐先按上下文和业务概念：

```text
supplychain
  trade
    domain
      order
      payment
  inventory
    domain
      stock
      reservation
      allocation
  warehouse
    domain
      shipment
      picking
  aftersale
    domain
      returnorder
      refund
```

这样看包结构就能看出业务结构，而不是只看到技术分层。

### 模块重构判断

| 现象 | 可能问题 | 调整方向 |
|---|---|---|
| entity包里几十个类 | 技术分包过强 | 按聚合或业务概念拆 |
| common里有业务枚举 | 共享模型污染 | 移回具体上下文 |
| order包依赖warehouse内部类 | 上下文边界被打穿 | 改用事件或防腐层 |
| service包里全是Manager | 业务语言缺失 | 用业务能力命名 |
