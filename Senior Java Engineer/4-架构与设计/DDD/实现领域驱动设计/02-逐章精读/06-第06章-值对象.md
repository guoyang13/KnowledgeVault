# 第06章：值对象

> 本章目标：学会用不可变的业务值表达概念，减少实体滥用，让模型更清晰、更安全。

## 本章在全书中的位置

第5章讲实体，第6章讲值对象。初学者往往过度使用实体，而忽略值对象。值对象是 DDD 中最能快速改善代码质量的模式之一。

## 一句话理解

值对象没有独立身份，按值判断相等，通常不可变，表达一个完整业务概念。

## 本章精读路线

读值对象时要训练一个习惯：看到多个基础字段经常一起出现，就问它们是不是一个业务概念。

例如：

```text
amount + currency → Money
startDate + endDate → DateRange
province + city + detail → Address
value + unit → Quantity
```

值对象的学习重点不是“包装字段”，而是让概念自己保护规则。

## 值对象的建模收益

| 收益 | 说明 |
|---|---|
| 业务语义更明确 | `Money` 比 `BigDecimal` 更能表达金额 |
| 校验更集中 | 金额非负、币种一致放在 Money 内 |
| 参数更安全 | 避免把 userId 传给 orderId |
| 代码更可读 | 方法签名就是业务语言 |
| 并发更安全 | 不可变对象不会被偷偷修改 |
| 测试更简单 | 值对象可独立测试规则 |

## 从字段到值对象的识别方法

找值对象可以从这些信号开始：

- 多个字段总是一起出现。
- 多个字段一起校验。
- 多个字段一起展示。
- 业务人员会把它们当成一个概念。
- 代码里到处重复相同校验。
- 方法参数列表中基础类型过多。

示例：如果很多方法都有 `BigDecimal amount, String currency`，就应该考虑 `Money`。

## 值对象和枚举

有些概念不要急着用 enum。比如订单状态如果有复杂状态行为，可以建成状态对象或让实体封装状态流转。

适合 enum：

- 值集合稳定。
- 行为简单。
- 不需要复杂扩展。

适合值对象/类型对象：

- 状态携带额外规则。
- 不同状态行为差异明显。
- 需要配置化或扩展。

## 值对象的特征

| 特征 | 解释 |
|---|---|
| 度量或描述 | 描述一个实体或业务状态 |
| 不变性 | 创建后不修改，变化时整体替换 |
| 概念整体 | 多个字段组合成一个业务概念 |
| 可替换性 | 值相同就可以替换 |
| 值相等 | 按字段值判断相等 |
| 无副作用行为 | 方法返回新值，不偷偷修改自身 |

## 典型值对象

| 值对象 | 字段 | 行为 |
|---|---|---|
| Money | amount、currency | add、subtract、compare |
| Address | province、city、district、detail | format、validate |
| DateRange | start、end | contains、overlaps、duration |
| Quantity | value、unit | add、convert |
| Email | value | validate、domain |
| PhoneNumber | countryCode、number | mask、validate |

## 值对象解决什么问题

### 1. 避免基础类型泛滥

错误方式：

```java
public void pay(BigDecimal amount, String currency) {
}
```

问题：

- amount 和 currency 可以不匹配。
- 任何 BigDecimal 都能传进来。
- 货币规则散落在各处。

更好的方式：

```java
public void pay(Money money) {
}
```

Money 自己保护金额和币种规则。

### 2. 表达业务概念整体

地址不是四个字符串，而是一个业务概念：

```java
public record Address(
    String province,
    String city,
    String district,
    String detail
) {
    public Address {
        if (province == null || city == null || detail == null) {
            throw new IllegalArgumentException("address is incomplete");
        }
    }
}
```

### 3. 降低错误赋值

如果所有 ID 都是 `Long`，就可能把 `userId` 传给 `orderId`。

更安全：

```java
public record UserId(Long value) {}
public record OrderId(Long value) {}
```

编译期就能减少误传。

## 不变性

值对象建议不可变。变化时创建新对象：

```java
public record Money(BigDecimal amount, String currency) {
    public Money add(Money other) {
        if (!currency.equals(other.currency)) {
            throw new IllegalArgumentException("currency mismatch");
        }
        return new Money(amount.add(other.amount), currency);
    }
}
```

这样做的好处：

- 不会被其他对象偷偷修改。
- 更容易测试。
- 更适合并发环境。
- 更容易表达业务计算。

## 实体和值对象如何选择

| 判断问题 | 选择 |
|---|---|
| 是否有独立身份？ | 有则实体 |
| 值一样是否就是同一个？ | 是则值对象 |
| 是否有生命周期？ | 有则实体 |
| 是否只是描述另一个对象？ | 倾向值对象 |
| 修改时是否可整体替换？ | 倾向值对象 |

例子：

```text
订单 Order：实体
订单金额 Money：值对象
收货地址 Address：值对象
订单项 OrderItem：可能是实体，也可能是值对象，取决于是否需要身份和生命周期
```

## 持久化值对象

值对象可以有多种持久化方式：

| 方式 | 例子 | 适合场景 |
|---|---|---|
| 嵌入字段 | amount、currency 存订单表 | 简单值对象 |
| JSON序列化 | address_json | 结构变化少、查询少 |
| 独立表 | order_address | 值结构复杂 |
| 枚举映射 | status_code | 状态型值对象 |

注意：持久化方式不应该反过来破坏值对象建模。

## Java落地建议

Java 17+ 可以优先用 record 表达简单值对象：

```java
public record DateRange(LocalDate start, LocalDate end) {
    public DateRange {
        if (start == null || end == null) {
            throw new IllegalArgumentException("date range is incomplete");
        }
        if (end.isBefore(start)) {
            throw new IllegalArgumentException("end cannot be before start");
        }
    }

    public boolean contains(LocalDate date) {
        return !date.isBefore(start) && !date.isAfter(end);
    }
}
```

复杂值对象也可以用普通 class，但要保证不可变。

## 常见误区

- 所有对象都建成实体。
- 用 `String`、`BigDecimal`、`Long` 到处裸奔。
- 值对象可变，导致引用被偷偷修改。
- 值对象没有校验，只是字段包装。
- 因为数据库没有单独表，就不建值对象。
- 把值对象设计得过大，承担实体职责。

| amount + currency | Money | 金额非负、币种一致 |
| startTime + endTime | DateRange | 结束时间不能早于开始时间 |

## 深化精读补充

### 逐节精读抓手

| 阅读块 | 要抓住的问题 | 学习产出 |
|---|---|---|
| 值对象特征 | 什么对象没有身份、靠值相等 | 值对象候选清单 |
| 不变性 | 为什么变化时整体替换 | 不可变实现方式 |
| 概念整体 | 哪些字段应该合成一个概念 | 字段组合表 |
| 行为和相等性 | 值对象能不能有方法 | equals/hashCode或record |
| 持久化 | 值对象如何落库但不被表结构绑架 | 映射方案 |

### 供应链值对象示例

| 值对象 | 字段 | 保护的规则 | 使用位置 |
|---|---|---|---|
| Money | amount、currency | 金额非负、币种一致 | 订单金额、退款金额 |
| Quantity | value、unit | 数量不能为负、单位一致 | 库存、订单项 |
| SkuId | value | 非空、格式合法 | 商品、库存、仓储 |
| WarehouseId | value | 非空、仓库编码合法 | 库存、出库单 |
| DateRange | start、end | 结束不能早于开始 | 活动、账期、Sprint |
| Address | province、city、detail | 地址完整性 | 收货、退货 |

这些对象都不应该只是字段包装。比如 `Quantity.add()` 必须检查单位一致，`Money.add()` 必须检查币种一致。

### 从基础类型到值对象

粗糙方法签名：

```java
reserve(String skuCode, BigDecimal qty, String unit, String warehouseCode);
```

更有领域语义：

```java
reserve(SkuId skuId, Quantity quantity, WarehouseId warehouseId);
```

后者让错误更早暴露：你不能轻易把仓库编码传给 SKU，也不能把不同单位的数量直接相加。
