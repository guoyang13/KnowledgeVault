# 第13章：集成限界上下文

> 本章目标：把上下文映射图落到具体集成方案中，让上下文协作但不互相污染。

## 本章在全书中的位置

第3章画上下文映射，第13章讨论如何实现这些关系。它把战略设计拉回工程实践。

## 一句话理解

集成限界上下文的关键是既传递必要信息，又保护各自模型边界。

## 本章精读路线

读本章时，重点不是背集成方式，而是学会选择：

```text
什么时候同步调用？
什么时候发布事件？
什么时候批处理？
什么时候需要防腐层？
什么时候宁可重复实现也不共享？
```

集成设计的核心问题是耦合。所有集成都会产生耦合，本章要学的是把耦合放到可控位置。

## 集成决策表

| 问题 | 倾向方案 |
|---|---|
| 调用方必须立即得到结果吗？ | 同步 API |
| 只是通知一个已经发生的事实吗？ | 事件 |
| 数据量大且实时性不高吗？ | 批处理 |
| 外部模型会污染本地语言吗？ | 防腐层 |
| 多个下游都需要稳定能力吗？ | 开放主机服务 |
| 交换格式需要长期兼容吗？ | 发布语言 |
| 共享后会影响核心域演进吗？ | 避免共享内核 |

不要因为“解耦”就所有事情都事件化，也不要因为“简单”就所有事情都同步调用。

## 集成失败要提前设计

跨上下文集成一定要考虑失败：

| 失败点 | 处理思路 |
|---|---|
| 同步接口超时 | 超时控制、降级、重试、补偿 |
| 事件发送失败 | Outbox、本地消息表、事务消息 |
| 事件重复消费 | eventId 幂等 |
| 事件乱序 | 聚合版本、序号、状态校验 |
| 下游长期失败 | 死信队列、人工补偿 |
| 发布语言升级 | 版本号、兼容字段、灰度 |

这些虽然偏工程实现，但会反过来影响事件和上下文边界设计。

## 防腐层代码职责

一个成熟防腐层通常包含：

```text
Client：负责调用外部系统
DTO：承接外部协议
Translator：把外部语言翻译成本地语言
Facade：给应用层提供稳定入口
ExceptionMapper：转换外部错误
```

防腐层不是把外部字段复制一遍，而是把外部模型的坏味道挡在边界外。

## 集成方式对比

| 方式 | 适合场景 | 优点 | 风险 |
|---|---|---|---|
| 同步 API | 需要立即响应 | 简单直接 | 强耦合、超时、级联失败 |
| 领域/集成事件 | 事实传播、最终一致 | 解耦、适合跨上下文 | 调试、顺序、幂等更复杂 |
| 批处理 | 数据量大、实时性低 | 稳定、成本低 | 延迟高 |
| 共享库 | 稳定通用能力 | 复用方便 | 版本耦合、模型污染 |
| 共享数据库 | 短期兼容 | 接入快 | 强耦合，通常应避免 |
| 防腐层 | 外部模型不可信 | 保护本地模型 | 翻译成本 |

## 同步API

同步 API 适合：

- 用户请求必须立即得到结果。
- 上游能力稳定。
- 调用链不长。
- 失败可以明确处理。

风险：

- 下游可用性依赖上游。
- 超时和重试可能放大故障。
- 接口模型可能污染本地模型。

同步 API 也需要防腐层，不要直接使用外部 DTO。

## 事件集成

事件适合传播已经发生的事实：

```text
OrderPaid
→ InventoryReserved
→ ShipmentCreated
```

事件消费者只根据事件做自己的决策，不应该反向依赖发布方内部模型。

设计事件时要考虑：

- 事件名是否是业务过去式。
- 是否包含消费者必要信息。
- 是否隐藏内部实现细节。
- 是否支持幂等。
- 是否有版本号。

## 防腐层

防腐层是集成章节的核心武器。

```text
外部模型
→ Client
→ Translator
→ 本地领域概念
→ Application/Domain
```

示例包结构：

```text
trade
  infrastructure
    acl
      erp
        ErpClient
        ErpOrderDTO
        ErpOrderTranslator
```

领域层只接受本地概念：

```java
public record ExternalPaymentConfirmation(
    String paymentNo,
    OrderId orderId,
    Money paidAmount
) {
}
```

## 发布语言

发布语言是上下文对外承诺的稳定信息格式，可以是：

- REST API schema
- 事件 schema
- 文件格式
- GraphQL schema
- protobuf schema

发布语言不应该等同于内部领域对象。对外格式要考虑兼容性和演进。

## 开放主机服务

当一个上下文需要被多个下游使用，可以提供开放主机服务：

```text
身份访问上下文
→ User API / Token API / Permission API
→ 交易、仓储、售后等下游
```

它要求上游承担版本治理、文档、兼容性和稳定性责任。

## 共享内核谨慎使用

共享内核只适合非常小、非常稳定、共同维护的模型。

适合共享：

- 基础值对象，如 Money、DateRange，且语义完全一致。
- 稳定协议对象。

不适合共享：

- 核心域实体。
- 会频繁变化的业务规则。
- 不同上下文含义不同的枚举。

## 常见误区

- 为了省事共享数据库。
- 把上游 DTO 当本地域模型。
- 事件包含整个聚合快照。
- 所有集成都走同步调用。
- 没有幂等和重试设计。
- 共享内核越做越大。

### 供应链集成场景

订单支付后触发库存预占：

```text
交易上下文：
Order.markPaid()
→ OrderPaid

库存上下文：
消费 OrderPaid
→ InventoryReservationService.reserve()
→ InventoryReserved / InventoryReserveFailed

仓储上下文：
消费 InventoryReserved
→ ShipmentOrderFactory.createFromReservation()
→ ShipmentOrderCreated
```

这条链路的关键是：交易上下文不直接操作库存，库存上下文不直接创建仓储出库单。每个上下文只处理自己的模型。
