# 第02章：领域、子域和限界上下文

> 本章目标：学会把一个大业务拆成子域，并为不同模型建立清晰边界。

## 本章在全书中的位置

第2章正式进入战略设计。战略设计的重点不是类怎么写，而是先决定：

- 哪些业务能力最核心？
- 哪些能力只是辅助？
- 哪些模型必须隔离？
- 哪些词在不同上下文中有不同含义？

如果边界没有划好，后面设计实体和值对象时就会把多个语义混在一起。

## 一句话理解

领域是整体业务问题，子域是业务能力分区，限界上下文是一套模型语言生效的边界。

## 本章精读路线

本章要分清三层概念：

```text
领域 Domain：公司面对的整体问题空间
子域 Subdomain：问题空间中的业务能力划分
限界上下文 Bounded Context：模型语言的一致性边界
```

读的时候不要把它们混成一个东西：

- 子域偏业务能力划分。
- 限界上下文偏模型语义边界。
- 微服务偏部署和运行边界。

一个子域可能包含多个限界上下文，一个限界上下文也可能服务于某个子域的一部分。

## 子域和限界上下文的关系

| 情况 | 说明 | 例子 |
|---|---|---|
| 一个子域一个上下文 | 最理想、最容易理解 | 身份访问子域对应身份上下文 |
| 一个子域多个上下文 | 子域内部语义复杂 | 订单子域拆成交易订单、履约订单、售后订单 |
| 多个子域共用一个上下文 | 早期系统或小团队常见 | 后台管理系统同时处理用户、订单、报表 |
| 上下文跨子域 | 通常是边界混乱信号 | 一个 OrderContext 同时处理交易、库存、仓储 |

初学者容易把“业务模块”直接等同于限界上下文。更好的判断方式是看语言是否一致：同一个词在这个边界里是否只有一个意思。

## 识别核心域的方法

核心域不是凭感觉选出来的，可以用下面的问题判断：

| 问题 | 如果答案是“是” |
|---|---|
| 这个能力是否决定客户选择我们而不是竞争对手？ | 倾向核心域 |
| 这个能力做差会不会直接影响收入、成本或体验？ | 倾向核心域 |
| 市面通用产品是否很难满足？ | 倾向核心域 |
| 领域专家是否投入大量时间解释细节？ | 倾向核心域 |
| 规则是否经常变化且需要快速响应？ | 倾向核心域 |

反过来，如果一个能力很重要但行业里高度标准化，比如登录、短信、文件上传，通常更接近通用子域。

## 限界上下文命名

上下文命名要使用业务语言，而不是技术语言。

不好的命名：

```text
BaseContext
CommonContext
DataContext
OrderServiceContext
```

更好的命名：

```text
TradeContext
InventoryContext
FulfillmentContext
AfterSalesContext
BillingContext
```

名字应该让业务专家大概能理解这个上下文负责什么。

## 核心概念

| 概念 | 解释 | 判断问题 |
|---|---|---|
| 领域 | 软件要解决的业务范围 | 这个系统服务哪个业务？ |
| 子域 | 领域中的能力区块 | 这个能力是否能独立讨论？ |
| 核心域 | 最影响竞争力的子域 | 做不好是否影响业务成败？ |
| 支撑子域 | 支撑核心域的专用能力 | 它重要但不形成差异化吗？ |
| 通用子域 | 行业内通用能力 | 是否能采购、复用或简单实现？ |
| 限界上下文 | 模型和语言的一致性边界 | 这个边界内同一个词是否只有一个含义？ |

## 子域划分

以供应链为例：

| 子域 | 类型 | 原因 |
|---|---|---|
| 需求预测 | 核心域 | 影响备货、库存和成本，是竞争力来源 |
| 库存分配 | 核心域 | 影响履约效率和缺货率 |
| 仓储作业 | 支撑子域 | 很重要，但通常服务于履约主流程 |
| 结算对账 | 支撑子域 | 保证业务闭环 |
| 用户权限 | 通用子域 | 多数系统都有成熟方案 |
| 消息通知 | 通用子域 | 可复用通用能力 |

核心域不是“最难写的代码”，而是“业务最不能输的地方”。

## 限界上下文为什么重要

复杂业务里，同一个词在不同场景下含义经常不同。

例子：“订单”：

| 上下文 | 订单的含义 | 关注点 |
|---|---|---|
| 交易上下文 | 买卖契约 | 商品、价格、优惠、支付 |
| 库存上下文 | 库存占用依据 | SKU、数量、锁定、释放 |
| 仓储上下文 | 出库任务来源 | 拣货、复核、发货 |
| 售后上下文 | 退货退款依据 | 责任、补偿、逆向物流 |

这些模型不应该强行共用一个 `Order` 类。

## 限界上下文不是微服务

| 对比 | 限界上下文 | 微服务 |
|---|---|---|
| 本质 | 模型语言边界 | 部署和运行边界 |
| 关注点 | 语义一致性 | 独立部署、伸缩、自治 |
| 是否必须一一对应 | 不必须 | 不必须 |
| 先后关系 | 通常先识别上下文 | 再决定服务拆分 |

一个限界上下文可以由多个服务实现，一个服务也可能暂时承载多个上下文。但长期看，模型边界和服务边界越一致，系统越容易演进。

## 识别限界上下文的信号

- 同一个词在不同团队中含义不同。
- 一个实体字段越来越多，很多字段只在某些流程中使用。
- 一个 service 同时处理多个业务视角。
- 一个改动经常牵动多个团队。
- 接口 DTO 越来越像“万能对象”。
- 不同业务能力的发布节奏不同。
- 业务专家无法就某个词达成一致。

## Java项目落地

错误方式：

```text
com.example
  controller
  service
  mapper
  entity
```

更好的上下文优先方式：

```text
com.example.supplychain
  trade
    application
    domain
    infrastructure
  inventory
    application
    domain
    infrastructure
  warehouse
    application
    domain
    infrastructure
  aftersale
    application
    domain
    infrastructure
```

这样做的目的不是追求目录漂亮，而是防止一个模型污染另一个模型。

## 常见误区

- 把组织部门直接等同于子域。
- 把数据库 schema 直接等同于限界上下文。
- 把微服务拆分当成限界上下文识别。
- 为了复用强行共享一个大模型。
- 只按技术层拆包，不按业务语言拆包。

### 供应链子域拆解示例

以“订单到履约”为例，不要直接把系统叫“订单系统”。可以先拆成：

| 子域 | 核心问题 | 类型判断 |
|---|---|---|
| 交易订单 | 买卖关系是否成立，金额和支付是否正确 | 支撑/核心，取决于业务 |
| 库存 | 是否有货，如何预占、扣减、释放 | 通常核心 |
| 仓储履约 | 如何拣货、复核、出库、发货 | 核心或支撑 |
| 售后 | 如何退货、退款、补偿 | 支撑，但规则复杂 |
| 结算 | 钱如何对齐、账如何平 | 支撑 |
| 通知 | 如何触达用户和系统 | 通用 |

核心域不是固定答案。对一家以履约效率取胜的公司，库存和仓储可能是核心域；对一家以营销交易取胜的公司，交易和定价可能更核心。

### 限界上下文识别示例

“库存”至少可能有这些语义：

| 上下文 | 库存含义 | 不能混淆的点 |
|---|---|---|
| 交易上下文 | 用户看到的可售能力 | 不等于仓库真实库存 |
| 库存上下文 | SKU 在仓/渠道的可用、预占、冻结 | 要保护数量不变量 |
| 仓储上下文 | 货位、批次、实物数量 | 关注作业和实物移动 |
| 财务上下文 | 库存金额和成本 | 关注价值而非数量作业 |

如果这些语义共用一个 `Stock` 模型，后续一定会混乱。
