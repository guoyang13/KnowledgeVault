# 第12章：资源库

> 本章目标：理解 Repository 是面向聚合的集合式接口，不是 DAO 的新名字。

## 本章在全书中的位置

第10章讲聚合，第12章讲如何保存和获取聚合。资源库是领域模型和持久化机制之间的重要边界。

## 一句话理解

资源库像一个聚合集合，负责按业务意图获取和保存聚合根，隐藏数据库细节。

## 本章精读路线

读 Repository 时，一定要把它和 Mapper/DAO 分开。Mapper 关心表，Repository 关心聚合。

重点理解：

1. Repository 接口属于领域语言。
2. Repository 实现属于基础设施。
3. Repository 通常以聚合根为单位。
4. 复杂查询不一定放 Repository。
5. Repository 不应该暴露持久化细节。

如果一个 Repository 的方法名全是数据库字段组合，它大概率退化成 DAO。

## Repository方法命名

更好的命名：

```java
Optional<Order> findById(OrderId id);
Optional<Order> findUnpaidOrder(OrderId id);
void save(Order order);
```

需要警惕的命名：

```java
List<OrderDO> selectByStatusAndCreateTime(...)
int updateStatusById(...)
Map<String, Object> queryOrderMap(...)
```

这些方法可能属于 Mapper 或 QueryService，而不是领域 Repository。

## 命令侧和查询侧分离

命令侧需要聚合：

```text
加载 Order 聚合
→ 调用 order.pay()
→ 保存 Order 聚合
```

查询侧需要视图：

```text
订单列表页
→ OrderListQueryService
→ OrderListDTO
```

不要为了页面查询，把聚合设计成巨大的数据视图。

## 映射成本的取舍

领域对象和持久化对象分离会增加映射代码，但能保护模型。

适合分离：

- 核心域。
- 聚合规则复杂。
- 数据库结构不干净。
- 需要避免 ORM 注解污染。

可以简化：

- 通用子域。
- 简单 CRUD。
- 规则很少。

DDD 不是要求所有地方都重，但核心域要认真保护。

## Repository和DAO区别

| 维度 | DAO | Repository |
|---|---|---|
| 面向对象 | 表、记录、SQL | 聚合根 |
| 语言 | 数据访问语言 | 领域语言 |
| 返回值 | DO、PO、Map、DTO | 聚合根 |
| 查询 | 字段组合查询 | 业务意图查询 |
| 位置 | 基础设施层 | 接口在领域层，实现通常在基础设施层 |
| 目标 | 操作数据 | 保存和重建模型 |

## Repository接口设计

领域层定义接口：

```java
public interface OrderRepository {
    Optional<Order> findById(OrderId id);
    void save(Order order);
}
```

基础设施层实现：

```java
public class MyBatisOrderRepository implements OrderRepository {
    private final OrderMapper mapper;

    @Override
    public Optional<Order> findById(OrderId id) {
        OrderDO orderDO = mapper.selectById(id.value());
        return Optional.ofNullable(orderDO).map(this::toDomain);
    }

    @Override
    public void save(Order order) {
        mapper.upsert(toDataObject(order));
    }
}
```

领域层不应该知道 MyBatis 或 JPA 细节。

## 资源库不是万能查询器

危险接口：

```java
List<Order> findByUserIdAndStatusAndCreatedTimeBetweenAndSkuCode(...);
```

这种接口可能是页面查询需求，不一定应该放到聚合资源库。

复杂查询可以走读模型：

```text
query
  OrderQueryService
  OrderListDTO
  OrderReportMapper
```

命令侧 Repository 负责聚合，查询侧可以为了页面和报表优化。

## Repository的粒度

通常一个聚合根对应一个 Repository。

```text
Order 聚合 → OrderRepository
Payment 聚合 → PaymentRepository
Inventory 聚合 → InventoryRepository
```

不要为聚合内部实体单独创建 Repository，否则外部可能绕过聚合根修改内部对象。

## 持久化和领域模型转换

基础设施层需要完成映射：

```text
Domain Aggregate
↔ Data Object / ORM Entity
↔ Database
```

有两种风格：

1. 领域对象直接由 ORM 管理。
2. 领域对象和持久化对象分离，手写映射。

前者省事但容易污染领域模型；后者更清晰但代码更多。复杂核心域更适合分离。

## 事务边界

应用服务通常控制事务：

```java
@Transactional
public void submit(SubmitOrderCommand command) {
    Order order = orderRepository.findById(command.orderId()).orElseThrow();
    order.submit();
    orderRepository.save(order);
    eventPublisher.publish(order.pullEvents());
}
```

聚合保证业务规则，Repository 保存聚合，应用服务控制事务。

## 常见误区

- Repository 只是 Mapper/DAO 改名。
- 为每张表创建 Repository。
- 为聚合内部实体创建 Repository。
- Repository 暴露大量任意查询。
- 领域层直接依赖 ORM 注解和数据库字段。
- 为了查询方便破坏聚合边界。

### 供应链Repository示例

库存预占聚合的 Repository：

```java
public interface InventoryReservationRepository {
    Optional<InventoryReservation> findById(ReservationId reservationId);
    Optional<InventoryReservation> findByOrderId(OrderId orderId);
    void save(InventoryReservation reservation);
}
```

不建议在这里放：

```java
List<InventoryReservation> findBySkuAndWarehouseAndStatusAndCreatedTime(...);
```

这更像后台列表查询，应该放到查询服务或读模型中。

### Repository和Mapper协作

```text
Application Service
→ InventoryReservationRepository
→ MyBatisInventoryReservationRepository
→ InventoryReservationMapper
→ inventory_reservation 表
```

领域层只知道 Repository 接口，不知道 Mapper、SQL、表结构。
