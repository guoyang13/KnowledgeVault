# 第10章：聚合

> 本章目标：理解聚合是事务一致性边界，学会先找业务不变量，再决定对象边界。

## 本章在全书中的位置

聚合是 DDD 战术设计中最难、也最容易误解的模式。实体和值对象回答“对象是什么”，聚合回答“哪些对象必须一起保持一致”。

## 一句话理解

聚合是一组对象的一致性边界，外部只能通过聚合根修改内部状态。

## 本章精读路线

聚合这一章要慢读。不要从“对象包含关系”入手，要从“业务规则必须怎样一致”入手。

推荐顺序：

1. 找业务命令：用户想做什么。
2. 找不变量：这个动作前后哪些规则必须成立。
3. 找一致性范围：哪些对象必须同事务更新。
4. 找聚合根：谁负责保护这些规则。
5. 找跨聚合协作：哪些规则可以事件最终一致。

如果跳过第2步，聚合设计基本都会退化成数据库外键模型。

## 聚合设计的三类问题

| 问题 | 说明 | 例子 |
|---|---|---|
| 业务规则问题 | 哪些规则必须立即正确 | 订单总金额必须等于订单项合计 |
| 并发问题 | 谁会同时修改同一对象 | 多用户同时预占库存 |
| 生命周期问题 | 哪些对象一起创建、变化、消亡 | 订单项通常随订单生命周期存在 |

聚合边界要同时考虑这三类问题。

## 聚合大小的权衡

| 聚合大小 | 优点 | 代价 |
|---|---|---|
| 大聚合 | 强一致规则容易维护 | 并发冲突、加载成本、事务范围大 |
| 小聚合 | 并发好、性能好、边界清楚 | 需要事件、补偿、最终一致 |

DDD 更倾向小聚合，但前提是你能接受最终一致并设计好事件协作。

## 聚合设计反例分析

反例：

```text
Order 聚合包含 Payment、InventoryReservation、Shipment
```

看似完整，实际问题是：

- Payment 可能属于支付上下文。
- InventoryReservation 可能属于库存上下文。
- Shipment 可能属于仓储上下文。
- 一个订单支付动作会锁住过多对象。

更合理：

```text
OrderPaid
→ InventoryReservation 独立处理
→ Shipment 独立处理
```

聚合边界不是为了让对象树完整，而是为了让规则和事务合理。

## 聚合不是对象树

初学者容易把聚合理解成：

```text
Order 包含 OrderItem，所以 Order 是聚合。
```

这只说对了一小部分。聚合真正关心的是：

- 哪些规则必须立即一致？
- 哪些对象必须在同一事务中修改？
- 外部应该通过哪个入口修改？
- 哪些对象可以最终一致？

## 聚合保护不变量

不变量是业务上任何时候都必须成立的规则。

例子：

| 聚合 | 不变量 |
|---|---|
| Order | 已支付订单不能修改商品；订单总金额等于订单项合计 |
| Inventory | 预占数量不能超过可用数量 |
| Sprint | 已关闭 Sprint 不能新增任务 |
| Account | 余额不能低于允许透支额度 |

设计聚合时，先找不变量，再找对象。

## 聚合根

聚合根是外部访问聚合的唯一入口。

规则：

- 外部只能持有聚合根引用。
- 外部不能直接修改聚合内部实体。
- 聚合根负责维护聚合内不变量。
- Repository 通常以聚合根为单位保存和获取。

## 聚合设计原则

| 原则 | 解释 |
|---|---|
| 聚合尽量小 | 大聚合会带来并发、事务和性能问题 |
| 一个事务修改一个聚合 | 跨聚合协作优先用领域事件 |
| 聚合之间通过 ID 引用 | 避免对象图无限扩大 |
| 通过聚合根访问内部对象 | 防止绕过业务规则 |
| 聚合边界服务不变量 | 不要为了查询方便扩大聚合 |

## 错误的大聚合

```text
Customer
  Orders
    OrderItems
      Product
  Payments
  Addresses
```

这种设计可能导致：

- 加载成本过高。
- 并发冲突严重。
- 一个小改动锁住大范围对象。
- 事务边界过大。

更好的方式通常是多个聚合通过 ID 协作：

```text
Customer
Order(customerId)
Payment(orderId)
Product(productId)
```

## 跨聚合一致性

如果订单支付后要扣库存，是否把订单和库存放在同一个聚合？

通常不应该。可以：

```text
Order.markPaid()
→ 发布 OrderPaid
→ Inventory 处理事件并预占/扣减库存
→ 发布 InventoryReserved
```

订单和库存属于不同一致性边界，使用最终一致更合理。

## Java聚合示例

```java
public class Order {
    private final OrderId id;
    private final List<OrderItem> items;
    private OrderStatus status;
    private Money totalAmount;

    public void addItem(ProductId productId, Quantity quantity, Money price) {
        if (status != OrderStatus.DRAFT) {
            throw new IllegalStateException("only draft order can add item");
        }
        items.add(new OrderItem(productId, quantity, price));
        recalculateTotal();
    }

    public void pay() {
        if (status != OrderStatus.SUBMITTED) {
            throw new IllegalStateException("only submitted order can be paid");
        }
        this.status = OrderStatus.PAID;
    }

    private void recalculateTotal() {
        this.totalAmount = items.stream()
            .map(OrderItem::subtotal)
            .reduce(Money.zero("CNY"), Money::add);
    }
}
```

外部不能直接改 `OrderItem`，否则订单总金额规则会失控。

## 聚合边界判断表

| 问题 | 如果答案是“是” |
|---|---|
| 是否必须强一致？ | 倾向同一聚合 |
| 是否只是查询方便？ | 不应放进同一聚合 |
| 是否有独立生命周期？ | 倾向独立聚合 |
| 是否高频并发修改？ | 倾向拆小 |
| 是否可通过事件最终一致？ | 倾向独立聚合 |

## 常见误区

- 按数据库外键设计聚合。
- 聚合越大越完整。
- 为查询方便把对象都放进一个聚合。
- 聚合之间直接持有完整对象。
- 一个事务修改多个聚合。
- 聚合根只是一个数据容器，没有业务方法。

### 供应链聚合示例

库存上下文中可以有 `InventoryReservation` 聚合：

```text
聚合根：InventoryReservation
内部值对象：SkuId、WarehouseId、Quantity
状态：CREATED、CONFIRMED、RELEASED、DEDUCTED
不变量：
- 预占数量必须大于0
- 已释放的预占不能再次扣减
- 已扣减的预占不能再次释放
- 同一个reservationId只能处理一次
事件：
- InventoryReserved
- InventoryReservationReleased
- InventoryDeducted
```

注意：它不应该直接包含 `Order` 聚合。它只需要保存 `orderId` 作为外部引用。

### 聚合边界反复追问

设计聚合时对每个内部对象追问：

```text
它是否必须和聚合根同事务变化？
它是否有独立生命周期？
外部是否需要单独引用它？
它是否只是为了查询方便被放进来？
如果短暂不一致，业务是否可以接受？
```

如果只是为了查询方便，就不应该扩大聚合，而应该用读模型。
