# DDD术语卡片

## 这个文件的作用

这个文件是速查卡，不是完整教材。它适合三种场景：

- 阅读章节时忘了某个术语，回来快速查。
- 复习时遮住解释，只看术语，检查自己能不能讲出来。
- 读完一章后，把自己的例子补充到对应术语下。

真正理解术语要回到章节文件和练习文件。比如理解“聚合”，不要只看这里，还要读 [[10-第10章-聚合]]，再做 [[01-DDD落地练习-以项目管理SaaS为例]]。

## Domain 领域

一句话：业务问题空间。

判断：先问业务要解决什么问题，而不是系统有哪些表。

## Subdomain 子域

一句话：领域内部按业务能力划分出来的部分。

判断：这个能力是核心差异化、支撑能力，还是通用能力？

## Core Domain 核心域

一句话：最值得投入设计精力、最能形成竞争力的子域。

误区：技术最难的不一定是核心域，业务最关键的才是。

## Bounded Context 限界上下文

一句话：一套模型和一套语言生效的边界。

判断：同一个词在这个边界内是否只有一个清晰含义？

## Ubiquitous Language 通用语言

一句话：业务专家和开发团队共同使用的模型语言。

判断：会议、文档、代码、测试是否使用同一套词？

## Context Map 上下文映射图

一句话：描述多个限界上下文之间关系的地图。

判断：两个上下文之间是合作、上下游、防腐、共享，还是各做各的？

## Entity 实体

一句话：有稳定身份和生命周期的业务对象。

判断：属性变了，它还是不是同一个对象？

## Value Object 值对象

一句话：用值表达的不可变业务概念。

判断：两个对象值一样，是否就可以认为一样？

## Domain Service 领域服务

一句话：不自然属于某个实体或值对象的领域行为。

判断：它是不是业务语言的一部分？是否无状态？

## Domain Event 领域事件

一句话：业务上已经发生的重要事实。

判断：这件事发生后，其他模块、上下文或人是否需要知道？

## Module 模块

一句话：按业务语言组织领域对象的容器。

判断：包名是否表达业务概念，而不是只表达技术层。

## Aggregate 聚合

一句话：事务一致性边界。

判断：哪些对象必须在一次业务操作里保持规则一致？

## Aggregate Root 聚合根

一句话：聚合对外的唯一入口。

判断：外部代码是否只能通过它修改聚合内部对象？

## Factory 工厂

一句话：封装复杂创建过程，保护创建时的业务规则。

判断：创建对象是否有复杂规则，调用方是否不应知道细节？

## Repository 资源库

一句话：面向聚合的集合式获取和保存接口。

判断：调用方是在找聚合，还是在随意查表？

## Application Service 应用服务

一句话：编排一个业务用例的服务。

判断：它是否只负责流程、事务和调用，而不承载核心业务规则？

## Anti-Corruption Layer 防腐层

一句话：翻译外部模型，保护本地模型不被污染。

判断：外部系统的语言和数据结构是否会破坏本地通用语言？

## Published Language 发布语言

一句话：上下文之间公开交换信息的稳定语言。

判断：多个消费者是否需要理解同一套事件、DTO 或 API Schema？

## Open Host Service 开放主机服务

一句话：上游上下文对外提供稳定访问协议。

判断：是否有多个下游系统需要接入同一能力？

## CQRS

一句话：命令和查询职责分离。

判断：写模型和读模型是否差异很大，且读性能或读形态要求复杂？

## Event Sourcing 事件源

一句话：用事件序列而不是当前状态来保存聚合。

判断：业务是否非常关心完整历史，且事件重放能带来价值？

## DDD-Lite

一句话：只使用部分战术模式，却缺少战略设计和通用语言的 DDD 表面化实践。

判断：项目里是否只有 Entity、Service、Repository，却没有限界上下文和上下文映射？
